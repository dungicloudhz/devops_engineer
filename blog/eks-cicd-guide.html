EKS · Khái niệm

☸️ EKS là gì? — Core Concepts

Amazon EKS là dịch vụ Kubernetes được quản lý. AWS lo control plane (API Server, etcd, Scheduler), bạn chỉ cần quản lý worker nodes và deploy workloads.

🧠
Control Plane
AWS quản lý hoàn toàn. API Server, etcd, Controller Manager, Scheduler — multi-AZ, auto-heal. Bạn không thể SSH vào control plane.
🖥️
Worker Nodes
EC2 instances hoặc Fargate pods chạy workloads. Managed Node Groups tự động cập nhật AMI. Bạn kiểm soát instance type, spot/on-demand.
🔌
Add-ons
AWS Load Balancer Controller, EBS CSI Driver, CoreDNS, kube-proxy, VPC CNI — cài thêm để tích hợp với các dịch vụ AWS.

EKS vs tự cài Kubernetes

Tiêu chíEKS (Managed)Self-managed K8s
Control PlaneAWS quản lý, SLA 99.95%Tự cài và maintain
Upgrade1-click trong console, zero downtimeManual, phức tạp
HATích hợp sẵn, multi-AZCần tự setup
AWS IntegrationALB, RDS, IAM, ECR nativeCần config thêm
Chi phí$0.10/giờ/cluster + EC2Chỉ EC2, nhưng tốn time ops
Khi nào dùngProductionLearning / On-prem

Các khái niệm K8s cần biết trước khi làm EKS

📦
Pod
Đơn vị nhỏ nhất trong K8s. Chứa 1 hoặc nhiều containers chia sẻ network và storage. Pods là ephemeral — có thể bị restart bất cứ lúc nào.
🔄
Deployment
Quản lý replicas của Pods. Định nghĩa số lượng, rolling update strategy. Đây là resource bạn sẽ dùng nhiều nhất.
🌐
Service
Stable endpoint cho Pods. ClusterIP (nội bộ), NodePort (expose qua node), LoadBalancer (AWS ALB/NLB).
⚙️
ConfigMap & Secret
ConfigMap lưu config không nhạy cảm. Secret lưu password, API keys — được base64 encode (không encrypt, dùng AWS Secrets Manager cho production).
EKS · Tạo Cluster

🛠️ Tạo EKS Cluster với eksctl

eksctl là công cụ CLI chính thức từ AWS để tạo và quản lý EKS clusters. Nhanh hơn nhiều so với console hoặc CloudFormation thủ công.

Cài đặt tools cần thiết

bash — cài aws-cli, eksctl, kubectl, helm
# ── AWS CLI v2 ──
curl "https://awscli.amazonaws.com/awscli-exe-linux-x86_64.zip" -o awscliv2.zip
unzip awscliv2.zip && sudo ./aws/install
aws configure   # nhập Access Key ID, Secret, region (ap-southeast-1)

# ── eksctl ──
ARCH=amd64
PLATFORM=$(uname -s)_$ARCH
curl -sLO "https://github.com/eksctl-io/eksctl/releases/latest/download/eksctl_${PLATFORM}.tar.gz"
tar -xzf eksctl_${PLATFORM}.tar.gz -C /tmp && sudo mv /tmp/eksctl /usr/local/bin
eksctl version   # 0.174.0

# ── kubectl ──
curl -LO "https://dl.k8s.io/release/$(curl -L -s https://dl.k8s.io/release/stable.txt)/bin/linux/amd64/kubectl"
chmod +x kubectl && sudo mv kubectl /usr/local/bin
kubectl version --client

# ── helm ──
curl https://raw.githubusercontent.com/helm/helm/main/scripts/get-helm-3 | bash

Tạo cluster bằng config file (khuyến nghị)

cluster.yaml — EKS cluster config đầy đủ
apiVersion: eksctl.io/v1alpha5
kind: ClusterConfig

metadata:
  name: prod-cluster
  region: ap-southeast-1     # Singapore
  version: "1.29"

# ── Availability Zones ──
availabilityZones: ["ap-southeast-1a", "ap-southeast-1b", "ap-southeast-1c"]

# ── VPC config ──
vpc:
  cidr: 10.0.0.0/16
  nat:
    gateway: HighlyAvailable  # 1 NAT GW mỗi AZ
  clusterEndpoints:
    publicAccess:  true
    privateAccess: true

# ── Managed Node Groups ──
managedNodeGroups:
  # On-demand nodes cho production workloads
  - name: ng-prod
    instanceType: t3.medium      # 2 vCPU, 4 GB RAM
    minSize: 2
    maxSize: 10
    desiredCapacity: 3
    volumeSize: 30               # GB, EBS gp3
    privateNetworking: true     # nodes trong private subnet
    labels:
      role: production
    tags:
      Environment: production
      k8s.io/cluster-autoscaler/enabled: "true"
    iam:
      withAddonPolicies:
        autoScaler:           true
        albIngress:           true
        cloudWatch:           true
        ebs:                  true
        efs:                  true

  # Spot nodes cho batch / non-critical workloads (rẻ hơn 70%)
  - name: ng-spot
    instanceTypes: [t3.medium, t3a.medium, m5.large]
    spot: true
    minSize: 0
    maxSize: 20
    desiredCapacity: 2
    labels:
      role: spot
    taints:
      - key: spot
        value: "true"
        effect: NoSchedule    # pods phải explicitly tolerate spot

# ── Add-ons ──
addons:
  - name: vpc-cni
    version: latest
  - name: coredns
    version: latest
  - name: kube-proxy
    version: latest
  - name: aws-ebs-csi-driver
    version: latest

# ── CloudWatch logging ──
cloudWatch:
  clusterLogging:
    enableTypes: ["api", "audit", "authenticator"]

EKS · Node Groups

🖥️ Node Groups & AWS Fargate

EKS hỗ trợ 3 loại compute: Managed Node Groups (EC2 bạn quản lý), Self-managed Nodes, và Fargate (serverless).

🔧
Managed Node Groups
AWS tự động provision EC2, cài kubelet, update AMI, drain trước khi terminate. Bạn chọn instance type và scaling policy. Phổ biến nhất cho production.
☁️
Fargate Profiles
Serverless — không cần quản lý EC2. AWS tự allocate compute cho từng Pod. Tốt cho batch jobs, dev environments. Không hỗ trợ DaemonSets.
💰
Spot Instances
Giảm chi phí 60–90%. Phù hợp workloads chịu được interruption: batch processing, stateless services với nhiều replicas.

Thêm Fargate Profile cho dev namespace

bash — tạo Fargate profile
eksctl create fargateprofile \
  --cluster  prod-cluster \
  --region   ap-southeast-1 \
  --name     fp-dev \
  --namespace dev \
  --labels   env=dev

# Pods trong namespace "dev" với label env=dev → chạy trên Fargate
# Không cần tạo EC2 node group riêng cho dev!

Cluster Autoscaler — tự động thêm/bớt nodes

bash — deploy Cluster Autoscaler
# Deploy Cluster Autoscaler với Helm
helm repo add autoscaler https://kubernetes.github.io/autoscaler
helm install cluster-autoscaler autoscaler/cluster-autoscaler \
  --namespace kube-system \
  --set autoDiscovery.clusterName=prod-cluster \
  --set awsRegion=ap-southeast-1 \
  --set rbac.serviceAccount.annotations."eks\.amazonaws\.com/role-arn"=arn:aws:iam::123456789:role/AmazonEKSClusterAutoscalerRole

# Verify
kubectl -n kube-system logs -f deployment/cluster-autoscaler
EKS · Security

🔐 IAM & IRSA — Phân quyền đúng cách

IRSA (IAM Roles for Service Accounts) cho phép Pods có IAM permissions mà không cần hardcode credentials. Đây là cách bảo mật chuẩn cho EKS.

Pod
K8s ServiceAccount
→
OIDC Provider
EKS built-in
→
IAM Role
AssumeRole
→
AWS Services
S3, RDS, SQS...
bash — setup IRSA cho Spring Boot app truy cập S3
# 1. Enable OIDC provider cho cluster
eksctl utils associate-iam-oidc-provider \
  --cluster prod-cluster \
  --region  ap-southeast-1 \
  --approve

# 2. Tạo IAM Role và Service Account liên kết
eksctl create iamserviceaccount \
  --cluster           prod-cluster \
  --namespace         production \
  --name              order-service-sa \
  --attach-policy-arn arn:aws:iam::aws:policy/AmazonS3ReadOnlyAccess \
  --attach-policy-arn arn:aws:iam::aws:policy/AmazonSQSFullAccess \
  --approve

# eksctl tự tạo IAM Role và annotate ServiceAccount với role ARN
kubectl describe sa order-service-sa -n production
Annotations: eks.amazonaws.com/role-arn: arn:aws:iam::123456789:role/eksctl-prod-cluster-addon-iamserviceaccount-...

# 3. Dùng ServiceAccount trong Deployment
# Thêm vào spec.template.spec:
serviceAccountName: order-service-sa
# → Pod tự động có AWS credentials qua mounted token!
# Trong Java: DefaultAWSCredentialsProviderChain tự detect

aws-auth ConfigMap — cho phép IAM users/roles truy cập cluster

bash — thêm developer và CI/CD role vào cluster
# Thêm IAM user cho developer
eksctl create iamidentitymapping \
  --cluster  prod-cluster \
  --arn      arn:aws:iam::123456789:user/dev-john \
  --group    system:masters \
  --username dev-john

# Thêm IAM Role cho CodeBuild (để kubectl apply)
eksctl create iamidentitymapping \
  --cluster  prod-cluster \
  --arn      arn:aws:iam::123456789:role/CodeBuildDeployRole \
  --group    system:masters \
  --username codebuild

# Xem aws-auth configmap
kubectl get configmap aws-auth -n kube-system -o yaml

AWS Load Balancer Controller — Ingress với ALB

bash + yaml — cài ALB Controller và tạo Ingress
# ── Cài AWS Load Balancer Controller ──
helm repo add eks https://aws.github.io/eks-charts
helm install aws-load-balancer-controller eks/aws-load-balancer-controller \
  -n kube-system \

apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: order-service-ingress
  namespace: production
  annotations:
    kubernetes.io/ingress.class: alb
    alb.ingress.kubernetes.io/scheme: internet-facing
    alb.ingress.kubernetes.io/target-type: ip
    alb.ingress.kubernetes.io/certificate-arn: arn:aws:acm:...
    alb.ingress.kubernetes.io/listen-ports: '[{"HTTP": 80}, {"HTTPS": 443}]'
    alb.ingress.kubernetes.io/ssl-redirect: "443"
    alb.ingress.kubernetes.io/group.name: production-alb  # share ALB
spec:
  rules:
  - host: api.company.com
    http:
      paths:
      - path: /api/orders
        pathType: Prefix
        backend:
          service:
            name: order-service
            port: {number: 8080}
      - path: /api/payments
        pathType: Prefix

EKS · Deploy

📦 Deploy Spring Boot lên EKS

Ví dụ đầy đủ — Deployment, Service, HPA, ConfigMap, Secret cho một Spring Boot microservice.

k8s/order-service.yaml — manifest hoàn chỉnh
---
# ── Namespace ──
apiVersion: v1
kind: Namespace
metadata:
  name: production
  labels: {env: production}
---
# ── ConfigMap ──
apiVersion: v1
kind: ConfigMap
metadata:
  name: order-service-config
  namespace: production
data:
  SPRING_PROFILES_ACTIVE: production
  SERVER_PORT: "8080"
  SPRING_DATASOURCE_URL: jdbc:postgresql://rds-host.rds.amazonaws.com:5432/shopdb
---
# ── Secret (giá trị từ AWS Secrets Manager qua external-secrets) ──
apiVersion: external-secrets.io/v1beta1
kind: ExternalSecret
metadata:
  name: order-service-secrets
  namespace: production
spec:
  refreshInterval: 1h
  secretStoreRef: {name: aws-secrets-manager, kind: SecretStore}
  target: {name: order-service-secret}
  data:
    - {secretKey: DB_PASSWORD, remoteRef: {key: prod/order-service/db-password}}
    - {secretKey: JWT_SECRET,  remoteRef: {key: prod/order-service/jwt-secret}}
---
# ── Deployment ──
apiVersion: apps/v1
kind: Deployment
metadata:
  name: order-service
  namespace: production
  labels: {app: order-service}
spec:
  replicas: 3
  selector:
    matchLabels: {app: order-service}
  strategy:
    type: RollingUpdate
    rollingUpdate: {maxSurge: 1, maxUnavailable: 0}  # zero downtime
  template:
    metadata:
      labels: {app: order-service}
      annotations:
        prometheus.io/scrape: "true"
        prometheus.io/port: "8080"
        prometheus.io/path: /actuator/prometheus
    spec:
      serviceAccountName: order-service-sa   # IRSA
      containers:
      - name: order-service
        image: 123456789.dkr.ecr.ap-southeast-1.amazonaws.com/order-service:v2.1.0
        ports:
        - {containerPort: 8080}
        envFrom:
        - {configMapRef: {name: order-service-config}}
        - {secretRef:    {name: order-service-secret}}
        resources:
          requests: {cpu: 250m, memory: 512Mi}
          limits:   {cpu: 1,    memory: 1Gi}
        readinessProbe:
          httpGet: {path: /actuator/health/readiness, port: 8080}
          initialDelaySeconds: 20
          periodSeconds: 10
        livenessProbe:
          httpGet: {path: /actuator/health/liveness, port: 8080}
          initialDelaySeconds: 60
          periodSeconds: 20
---
# ── Service ──
apiVersion: v1
kind: Service
metadata:
  name: order-service
  namespace: production
spec:
  selector: {app: order-service}
  ports:
  - {port: 80, targetPort: 8080}
  type: ClusterIP
EKS · Auto Scaling

📈 HPA — Horizontal Pod Autoscaler

HPA tự động tăng/giảm số replicas dựa trên CPU, memory hoặc custom metrics từ Prometheus.

hpa.yaml — scale dựa trên CPU và custom metrics
# ── Cài Metrics Server trước ──
# kubectl apply -f https://github.com/kubernetes-sigs/metrics-server/releases/latest/download/components.yaml

apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: order-service-hpa
  namespace: production
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: order-service
  minReplicas: 3
  maxReplicas: 20
  metrics:
    # Scale khi CPU > 70%
  - type: Resource
    resource:
      name: cpu
      target:
        type: Utilization
        averageUtilization: 70
    # Scale khi Memory > 80%
  - type: Resource
    resource:
      name: memory
      target:
        type: Utilization
        averageUtilization: 80
  behavior:
    scaleUp:
      stabilizationWindowSeconds: 60   # đợi 60s mới scale up tiếp
      policies:
      - {type: Pods, value: 4, periodSeconds: 60}  # max 4 pods/phút
    scaleDown:
      stabilizationWindowSeconds: 300  # đợi 5 phút mới scale down
      policies:
      - {type: Pods, value: 1, periodSeconds: 60}
✅ Verify HPA
kubectl get hpa -n production — xem trạng thái HPA, targets và replicas hiện tại.
kubectl describe hpa order-service-hpa -n production — xem lịch sử scale events.

🔁 Pipeline CI/CD trên AWS

AWS cung cấp bộ công cụ CI/CD native tích hợp chặt với IAM, ECR, EKS: CodePipeline (orchestrator), CodeBuild (runner), CodeDeploy (deploy).

AWS Native CI/CD Pipeline
Source

→

mvn test, sonar
→
CodeBuild Build
docker build + push ECR
→
Approve
manual gate (opt)
→

🚀
CodePipeline
Orchestrate toàn bộ pipeline. Trigger từ GitHub/CodeCommit. Kết nối các stages. Notify qua SNS. Tích hợp với mọi AWS service.
🔨
CodeBuild
Managed build server. Chạy commands trong Docker container. Tự scale. Trả tiền theo phút. Tích hợp ECR, SSM, Secrets Manager.
🐳
ECR
Elastic Container Registry — private Docker registry của AWS. Tích hợp IAM, quét lỗ hổng bảo mật, lifecycle policies tự xóa image cũ.
🔄
ArgoCD
GitOps CD tool. Sync K8s manifests từ Git repo. Auto-heal khi cluster drift khỏi desired state. Audit trail đầy đủ.

🚀 CodePipeline — Tạo Pipeline

cloudformation/pipeline.yaml — CodePipeline + CodeBuild
AWSTemplateFormatVersion: '2010-09-09'
Description: CI/CD Pipeline for order-service

Parameters:
  GitHubOwner:
    Type: String
    Default: company
  GitHubRepo:
    Type: String
    Default: order-service
  EKSClusterName:
    Type: String
    Default: prod-cluster

Resources:
  ArtifactBucket:
    Type: AWS::S3::Bucket
    Properties:
      VersioningConfiguration: {Status: Enabled}

  Pipeline:
    Type: AWS::CodePipeline::Pipeline
    Properties:
      RoleArn: !GetAtt PipelineRole.Arn
      ArtifactStore:
        Type: S3
        Location: !Ref ArtifactBucket
      Stages:

        # Stage 1: Source từ GitHub
        - Name: Source
          Actions:
          - Name: GitHubSource
            ActionTypeId:
              Category: Source
              Owner: ThirdParty
              Provider: GitHub
              Version: '1'
            Configuration:
              Owner: !Ref GitHubOwner
              Repo:  !Ref GitHubRepo
              Branch: main
              OAuthToken: !Sub "{{resolve:secretsmanager:github-token:SecretString:token}}"
            OutputArtifacts: [{Name: SourceCode}]

        # Stage 2: Test
        - Name: Test
          Actions:
          - Name: UnitTest
            ActionTypeId:
              Category: Build
              Owner: AWS
              Provider: CodeBuild
              Version: '1'
            Configuration:
              ProjectName: !Ref TestProject
            InputArtifacts:  [{Name: SourceCode}]

        # Stage 3: Build Docker + Push ECR
        - Name: Build
          Actions:
          - Name: BuildAndPush
            ActionTypeId:
              Category: Build
              Owner: AWS
              Provider: CodeBuild
              Version: '1'
            Configuration:
              ProjectName: !Ref BuildProject
            InputArtifacts:  [{Name: SourceCode}]
            OutputArtifacts: [{Name: BuildOutput}]

        # Stage 4: Manual approval trước production
        - Name: Approve
          Actions:
          - Name: ManualApproval
            ActionTypeId:
              Category: Approval
              Owner: AWS
              Provider: Manual
              Version: '1'

        # Stage 5: Deploy lên EKS
        - Name: Deploy
          Actions:
          - Name: DeployToEKS
            ActionTypeId:
              Category: Build
              Owner: AWS
              Provider: CodeBuild
              Version: '1'
            Configuration:
              ProjectName: !Ref DeployProject
            InputArtifacts: [{Name: BuildOutput}]
CI/CD · CodeBuild

🔨 CodeBuild & buildspec.yml

File buildspec.yml định nghĩa chính xác những gì CodeBuild chạy trong mỗi giai đoạn.

buildspec-test.yml — chạy unit tests

buildspec-test.yml
version: 0.2

env:
  variables:
    JAVA_HOME: /usr/lib/jvm/java-21-amazon-corretto
  parameter-store:
    SONAR_TOKEN: /prod/sonar/token
    DB_URL: /prod/order-service/test-db-url

phases:
  install:
    runtime-versions:
      java: corretto21
    commands:
      - echo "Installing dependencies..."
      - mvn dependency:go-offline -q   # cache deps trước

  pre_build:
    commands:
      - echo "Pre-build phase..."
      - echo "Commit SHA $CODEBUILD_RESOLVED_SOURCE_VERSION"

  build:
    commands:
      - echo "Running tests..."
      - mvn clean test
      # SonarQube analysis
      - mvn sonar:sonar
          -Dsonar.projectKey=order-service
          -Dsonar.host.url=https://sonarcloud.io
          -Dsonar.login=$SONAR_TOKEN

  post_build:
    commands:
      - echo "Tests completed on `date`"

reports:
  test-reports:
    files: ["target/surefire-reports/**/*.xml"]
    file-format: JUNITXML
  coverage-reports:
    files: ["target/site/jacoco/jacoco.xml"]
    file-format: JACOCOXML

cache:
  paths:
    - '/root/.m2/**/*'  # cache Maven local repo → tăng tốc build lần sau

buildspec-build.yml — build Docker image và push ECR

buildspec-build.yml
version: 0.2

env:
  variables:
    ECR_REGISTRY: 123456789.dkr.ecr.ap-southeast-1.amazonaws.com
    IMAGE_REPO:   order-service
    AWS_REGION:   ap-southeast-1

phases:
  pre_build:
    commands:
      # Login vào ECR
      - aws ecr get-login-password --region $AWS_REGION | docker login --username AWS --password-stdin $ECR_REGISTRY
      # Tag image với commit SHA và "latest"
      - IMAGE_TAG=$CODEBUILD_RESOLVED_SOURCE_VERSION
      - echo "IMAGE_TAG=$IMAGE_TAG" >> /tmp/env.sh

  build:
    commands:
      - echo "Building Docker image..."
      # Multi-stage build — production image nhỏ, không có Maven, không có src
      - docker build
          --build-arg JAR_FILE=target/order-service.jar
          --cache-from $ECR_REGISTRY/$IMAGE_REPO:latest
          -t $ECR_REGISTRY/$IMAGE_REPO:$IMAGE_TAG
          -t $ECR_REGISTRY/$IMAGE_REPO:latest
          .
      - echo "Build completed at `date`"

  post_build:
    commands:
      - echo "Pushing images to ECR..."
      - docker push $ECR_REGISTRY/$IMAGE_REPO:$IMAGE_TAG
      - docker push $ECR_REGISTRY/$IMAGE_REPO:latest
      # Tạo imagedefinitions.json cho CodeDeploy
      - printf '[{"name":"order-service","imageUri":"%s"}]' $ECR_REGISTRY/$IMAGE_REPO:$IMAGE_TAG > imagedefinitions.json
      # Tạo imageTag.txt để build stage sau đọc
      - echo $IMAGE_TAG > imageTag.txt

artifacts:
  files:
    - imagedefinitions.json
    - imageTag.txt
    - k8s/**     # K8s manifests để deploy stage dùng

Dockerfile multi-stage — Spring Boot production image

Dockerfile
# ── Stage 1: Build ──
FROM eclipse-temurin:21-jdk-alpine AS builder
WORKDIR /app
COPY .mvn/   .mvn/
COPY mvnw    pom.xml  ./
RUN ./mvnw dependency:go-offline -q    # cache layer
COPY src ./src
RUN ./mvnw clean package -DskipTests

# ── Stage 2: Extract layers (Spring Boot layered jar) ──
FROM eclipse-temurin:21-jre-alpine AS extractor
WORKDIR /app
COPY --from=builder /app/target/*.jar app.jar
RUN java -Djarmode=layertools -jar app.jar extract

# ── Stage 3: Production image — nhỏ nhất ──
FROM eclipse-temurin:21-jre-alpine
WORKDIR /app

# Non-root user
RUN addgroup -S spring && adduser -S spring -G spring
USER spring:spring

# Copy layers (caching tốt hơn)
COPY --from=extractor /app/dependencies/          ./
COPY --from=extractor /app/spring-boot-loader/    ./
COPY --from=extractor /app/snapshot-dependencies/ ./
COPY --from=extractor /app/application/           ./

EXPOSE 8080
HEALTHCHECK --interval=30s --timeout=5s --retries=3 \
  CMD wget -qO- http://localhost:8080/actuator/health | grep UP || exit 1

ENTRYPOINT ["java", "-XX:+UseContainerSupport", "-XX:MaxRAMPercentage=75", "org.springframework.boot.loader.launch.JarLauncher"]
CI/CD · GitHub Actions

⚡ GitHub Actions → EKS

Thay thế CodePipeline bằng GitHub Actions — phổ biến hơn, dễ debug, OIDC authentication không cần AWS credentials.

.github/workflows/deploy-eks.yml
name: Deploy to EKS

on:
  push:
    branches: [main]
  pull_request:
    branches: [main]

env:
  AWS_REGION:   ap-southeast-1
  EKS_CLUSTER:  prod-cluster
  ECR_REGISTRY: 123456789.dkr.ecr.ap-southeast-1.amazonaws.com
  IMAGE_NAME:   order-service

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-java@v4
        with: {java-version: '21', distribution: temurin, cache: maven}
      - name: Run tests
        run: mvn clean test

  build-and-deploy:
    needs: test
    runs-on: ubuntu-latest
    if: github.ref == 'refs/heads/main'
    permissions:
      id-token: write   # cần cho OIDC
      contents: read

    steps:
      - uses: actions/checkout@v4

      # ── Auth AWS bằng OIDC (không cần hardcode credentials!) ──
      - name: Configure AWS credentials (OIDC)
        uses: aws-actions/configure-aws-credentials@v4
        with:
          role-to-assume: arn:aws:iam::123456789:role/GitHubActionsDeployRole
          aws-region: ${{ env.AWS_REGION }}

      - name: Login to Amazon ECR
        id: login-ecr
        uses: aws-actions/amazon-ecr-login@v2

      - name: Build, tag, push Docker image
        env:
          IMAGE_TAG: ${{ github.sha }}
        run: |
          docker build -t $ECR_REGISTRY/$IMAGE_NAME:$IMAGE_TAG .
          docker push $ECR_REGISTRY/$IMAGE_NAME:$IMAGE_TAG
          docker tag $ECR_REGISTRY/$IMAGE_NAME:$IMAGE_TAG $ECR_REGISTRY/$IMAGE_NAME:latest
          docker push $ECR_REGISTRY/$IMAGE_NAME:latest

      - name: Update kubeconfig for EKS
        run: aws eks update-kubeconfig --name $EKS_CLUSTER --region $AWS_REGION

      - name: Deploy to EKS
        env:
          IMAGE_TAG: ${{ github.sha }}
        run: |
          sed -i "s|IMAGE_TAG_PLACEHOLDER|$IMAGE_TAG|g" k8s/order-service.yaml
          kubectl apply -f k8s/ -n production
          kubectl rollout status deployment/order-service -n production --timeout=5m

      - name: Verify deployment
        run: kubectl get pods -n production -l app=order-service

bash + yaml — cài ArgoCD và tạo Application
# ── Cài ArgoCD ──
kubectl create namespace argocd
kubectl apply -n argocd -f https://raw.githubusercontent.com/argoproj/argo-cd/stable/manifests/install.yaml

# Expose ArgoCD UI qua LoadBalancer
kubectl patch svc argocd-server -n argocd -p '{"spec": {"type": "LoadBalancer"}}'

# Lấy admin password
kubectl -n argocd get secret argocd-initial-admin-secret \
  -o jsonpath="{.data.password}" | base64 -d

# ── ArgoCD Application ──
apiVersion: argoproj.io/v1alpha1
kind: Application
metadata:
  name: order-service
  namespace: argocd
spec:
  project: default
  source:
    repoURL: https://github.com/company/k8s-manifests
    targetRevision: HEAD
    path: production/order-service   # thư mục chứa K8s yamls
  destination:
    server: https://kubernetes.default.svc
    namespace: production
  syncPolicy:
    automated:
      prune:    true   # xóa resources không còn trong Git
      selfHeal: true   # tự fix khi ai đó kubectl edit thủ công
    syncOptions:
      - CreateNamespace=true
      - ServerSideApply=true
  ignoreDifferences:
  - group: apps
    kind: Deployment
    jsonPointers: [/spec/replicas]  # ignore HPA managed replicas

# ── CI update image tag và push lên infra repo ──
# Thêm vào GitHub Actions workflow sau khi push image:
run: |
  git clone https://github.com/company/k8s-manifests
  cd k8s-manifests
  sed -i "s|order-service:.*|order-service:$IMAGE_TAG|" production/order-service/deployment.yaml
  git commit -am "chore: update order-service image to $IMAGE_TAG"
  git push
# ArgoCD tự phát hiện change trong vài giây và sync!

🗺️ Checklist Production Readiness
  • EKS: Node group trải đều 3 AZs — cluster vẫn hoạt động khi 1 AZ xuống
  • EKS: IRSA cho mọi service cần AWS — không hardcode credentials trong code hoặc ConfigMap
  • EKS: Resource requests/limits cho mọi container — tránh 1 pod ăn hết RAM node
  • EKS: readinessProbe và livenessProbe đúng — rolling update không gây downtime
  • EKS: HPA cấu hình đúng — scale up trước khi bị quá tải, scale down chậm để tránh oscillation
  • CI/CD: OIDC auth cho GitHub Actions — không lưu AWS credentials trong Secrets
  • CI/CD: ECR lifecycle policy — giữ 30 images, tự xóa cũ tránh tốn tiền storage
  • CI/CD: Rollback plan — ArgoCD one-click rollback hoặc kubectl rollout undo
  • Lens: Cài Metrics (Prometheus) ngay để có charts CPU/RAM real-time
  • Lens: Bookmark thường xuyên xem pod status production sau mỗi deploy
