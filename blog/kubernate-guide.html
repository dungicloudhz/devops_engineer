OVERVIEW

Kiến trúc Kubernetes — Big Picture

Control Plane · Worker Nodes · Pods · Tại sao cần K8s?

Tại sao dùng Kubernetes?
🔄
Self-healing
Pod crash tự động restart. Node die tự reschedule sang node khác. Không cần thức đêm trực server.
📈
Auto Scaling
HPA tự scale pods khi CPU/RAM cao. Cluster Autoscaler tự thêm node khi cần. Scale về 0 khi idle.
🚀
Zero-downtime Deploy
Rolling update thay thế pods dần dần. Rollback nhanh chóng khi có lỗi. Blue-green, canary deployments.
⚙️
Declarative Config
Khai báo trạng thái mong muốn bằng YAML. K8s tự làm mọi thứ để đạt trạng thái đó.

POD

Pod — Đơn vị nhỏ nhất trong K8s

Một hoặc nhiều containers cùng network namespace, storage, lifecycle

Pod YAML đầy đủ với giải thích
pod.yaml — annotatedPod
apiVersion: v1
kind: Pod
metadata:
  name:      order-service-pod
  namespace: production
  labels:                           # selector matching dùng labels
    app:     order-service
    version: v2.1.0
    tier:    backend
  annotations:
    prometheus.io/scrape: "true"
    prometheus.io/port:   "8080"
spec:
  containers:
  - name:  order-service
    image: registry.company.com/order-service:v2.1.0
    ports:
    - containerPort: 8080
      name:          http
    env:
    - name:  SPRING_PROFILES_ACTIVE
      value: prod
    - name:  DB_PASSWORD
      valueFrom:
        secretKeyRef:               # lấy từ Secret
          name: order-service-secrets
          key:  db-password
    resources:
      requests:                       # tài nguyên tối thiểu (scheduler dùng)
        memory: "256Mi"
        cpu:    "250m"              # 250 millicores = 0.25 CPU
      limits:                         # tài nguyên tối đa (container bị kill nếu vượt)
        memory: "512Mi"
        cpu:    "1000m"
    livenessProbe:
      httpGet:
        path: /actuator/health/liveness
        port: 8080
      initialDelaySeconds: 60
      periodSeconds:       10
    readinessProbe:
      httpGet:
        path: /actuator/health/readiness
        port: 8080
      initialDelaySeconds: 30
      periodSeconds:       5
    volumeMounts:
    - name:      config-volume
      mountPath: /app/config
      readOnly:  true
  volumes:
  - name: config-volume
    configMap:
      name: order-service-config
  imagePullSecrets:
  - name: registry-credentials  # pull từ private registry
  restartPolicy: Always            # Always | OnFailure | Never
  terminationGracePeriodSeconds: 30
⚠️
Không tạo Pod trực tiếp trong production! Pod là ephemeral — nếu node chết, Pod không tự rebuild. Luôn dùng Deployment (hoặc StatefulSet, DaemonSet) để quản lý Pods. Pod tạo trực tiếp chỉ dùng để test/debug.
DEPLOYMENT

Deployment & ReplicaSet — Stateless Workloads

Quản lý rolling update, rollback, scaling cho stateless apps

Deployment YAML production-ready
deployment.yaml — order-service với rolling updateDeployment
apiVersion: apps/v1
kind: Deployment
metadata:
  name:      order-service
  namespace: production
  labels:
    app: order-service
spec:
  replicas: 3                         # chạy 3 pods song song
  selector:
    matchLabels:
      app: order-service
  strategy:
    type: RollingUpdate               # Recreate hoặc RollingUpdate
    rollingUpdate:
      maxSurge:       1               # tối đa thêm 1 pod trong khi update
      maxUnavailable: 0               # không cho pod nào unavailable (zero-downtime!)
  minReadySeconds: 10                 # chờ 10s sau khi pod ready rồi mới tiếp
  progressDeadlineSeconds: 600       # timeout rollout sau 10 phút
  template:                            # PodTemplate — giống Pod spec
    metadata:
      labels:
        app:     order-service
        version: v2.1.0
    spec:
      affinity:
        podAntiAffinity:               # pods không chạy cùng node
          preferredDuringSchedulingIgnoredDuringExecution:
          - weight: 100
            podAffinityTerm:
              labelSelector:
                matchLabels:
                  app: order-service
              topologyKey: kubernetes.io/hostname
      topologySpreadConstraints:       # phân tán đều qua availability zones
      - maxSkew:           1
        topologyKey:       topology.kubernetes.io/zone
        whenUnsatisfiable: DoNotSchedule
        labelSelector:
          matchLabels:
            app: order-service
      containers:
      - name:  order-service
        image: registry.company.com/order-service:v2.1.0
        ports:
        - containerPort: 8080
        envFrom:                         # load tất cả keys từ ConfigMap
        - configMapRef:
            name: order-service-config
        - secretRef:
            name: order-service-secrets
        resources:
          requests: {memory: "256Mi", cpu: "250m"}
          limits:   {memory: "512Mi", cpu: "1000m"}
        livenessProbe:
          httpGet: {path: /actuator/health/liveness, port: 8080}
          initialDelaySeconds: 60
          periodSeconds: 10
          failureThreshold: 3
        readinessProbe:
          httpGet: {path: /actuator/health/readiness, port: 8080}
          initialDelaySeconds: 30
          periodSeconds: 5
          failureThreshold: 3
        lifecycle:
          preStop:                       # graceful shutdown
            exec:
              command: ["/bin/sh", "-c", "sleep 5"]
      terminationGracePeriodSeconds: 60
Rollout commands
kubectl rollout — update, status, rollbackRollout
# Update image (trigger rolling update)
$ kubectl set image deployment/order-service order-service=registry.company.com/order-service:v2.2.0 -n production

# Xem trạng thái rollout
$ kubectl rollout status deployment/order-service -n production
Waiting for deployment "order-service" rollout to finish: 1 of 3 updated...
deployment "order-service" successfully rolled out

# Xem lịch sử rollout
$ kubectl rollout history deployment/order-service -n production

# Rollback về version trước
$ kubectl rollout undo deployment/order-service -n production

# Rollback về revision cụ thể
$ kubectl rollout undo deployment/order-service --to-revision=2 -n production

# Scale thủ công
$ kubectl scale deployment/order-service --replicas=5 -n production

# Pause/Resume rolling update
$ kubectl rollout pause deployment/order-service -n production
$ kubectl rollout resume deployment/order-service -n production

service.yaml — ClusterIP và LoadBalancerService
# ── ClusterIP (internal access) ──
apiVersion: v1
kind: Service
metadata:
  name:      order-service
  namespace: production
spec:
  type:     ClusterIP              # default
  selector:
    app: order-service            # match pods với label này
  ports:
  - name:       http
    port:       80                  # port của Service
    targetPort: 8080              # port trong container
    protocol:   TCP
# DNS: order-service.production.svc.cluster.local
# hoặc ngắn: order-service (trong cùng namespace)

---
# ── LoadBalancer (external access via cloud LB) ──
apiVersion: v1
kind: Service
metadata:
  name: api-gateway-lb
  annotations:
    service.beta.kubernetes.io/aws-load-balancer-type: "nlb"
    service.beta.kubernetes.io/aws-load-balancer-internal: "false"
spec:
  type: LoadBalancer
  selector:
    app: api-gateway
  ports:
  - port: 443
    targetPort: 8443

---
# ── Headless Service (cho StatefulSet, DNS per pod) ──
apiVersion: v1
kind: Service
metadata:
  name: postgres
spec:
  clusterIP: None                   # Headless — không có virtual IP
  selector:
    app: postgres
  ports:
  - port: 5432
# DNS: postgres-0.postgres.default.svc.cluster.local (StatefulSet)

    max:                          # maximum cho một container

CONFIG

ConfigMap & Secret — Quản lý cấu hình

Tách config khỏi code, quản lý secrets an toàn

ConfigMap — 3 cách mount
configmap.yaml + 3 cách dùngConfigMap
# ── Tạo ConfigMap ──
apiVersion: v1
kind: ConfigMap
metadata:
  name:      order-service-config
  namespace: production
data:
  # Key-value pairs
  APP_ENV:                production
  LOG_LEVEL:              INFO
  KAFKA_BOOTSTRAP_SERVERS:kafka:9092
  # File content
  application.yml: |
    spring:
      datasource:
        url: jdbc:postgresql://postgres:5432/shopdb
      kafka:
        bootstrap-servers: kafka:9092
    management:
      endpoints:
        web:
          exposure:
            include: "health,info,metrics,prometheus"

# ── Cách 1: envFrom (load tất cả keys thành env vars) ──
envFrom:
- configMapRef:
    name: order-service-config

# ── Cách 2: env.valueFrom (lấy từng key) ──
env:
- name:  LOG_LEVEL
  valueFrom:
    configMapKeyRef:
      name: order-service-config
      key:  LOG_LEVEL

# ── Cách 3: volumeMount (mount as files) ──
volumeMounts:
- name:      config-vol
  mountPath: /app/config
  readOnly:  true
volumes:
- name: config-vol
  configMap:
    name: order-service-config
    items:                          # chỉ mount một số keys
    - key:  application.yml
      path: application.yml         # → /app/config/application.yml
Secret — Mã hóa và mount an toàn
secret.yaml — tạo và mount secretsSecret
# ── Tạo Secret (values phải base64 encode) ──
apiVersion: v1
kind: Secret
metadata:
  name:      order-service-secrets
  namespace: production
type: Opaque
data:
  # echo -n "secret_password" | base64
  db-password:    c2VjcmV0X3Bhc3N3b3Jk
  redis-password: cmVkaXNfc2VjcmV0
stringData:                         # hoặc dùng stringData (K8s tự encode)
  jwt-secret: my-super-secret-jwt-key

# ── Tạo Secret từ kubectl (cách an toàn hơn) ──
$ kubectl create secret generic order-service-secrets \
    --from-literal=db-password=secret \
    --from-literal=redis-password=redis_pass \
    --from-file=tls.crt=./certs/tls.crt \
    -n production

# ── TLS Secret ──
$ kubectl create secret tls my-tls-secret \
    --cert=tls.crt --key=tls.key -n production

# ── ImagePullSecret (private registry) ──
$ kubectl create secret docker-registry registry-credentials \
    --docker-server=registry.company.com \
    --docker-username=robot-ci \
    --docker-password=$TOKEN \
    -n production
⚠️
Secret không thực sự "secret" theo mặc định! K8s Secret chỉ base64 encode, không encrypt. Cần bật Encryption at Rest cho etcd, hoặc dùng Sealed Secrets, HashiCorp Vault, AWS Secrets Manager cho production thực sự.

Volumes, PV & PVC — Persistent Storage

PersistentVolume, PersistentVolumeClaim, StorageClass

StorageClass → PVC → Pod
storageclass.yaml + pvc.yaml + pod mountStorage
# ── StorageClass (cloud provider cấp storage) ──
apiVersion: storage.k8s.io/v1
kind: StorageClass
metadata:
  name: fast-ssd
provisioner: ebs.csi.aws.com        # AWS EBS
parameters:
  type:      gp3
  encrypted: "true"
reclaimPolicy:    Retain             # Retain | Delete | Recycle
volumeBindingMode: WaitForFirstConsumer
allowVolumeExpansion: true

---
# ── PersistentVolumeClaim — yêu cầu storage ──
apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name:      postgres-data
  namespace: production
spec:
  accessModes:
  - ReadWriteOnce                    # RWO | ROX | RWX | RWOP
  storageClassName: fast-ssd
  resources:
    requests:
      storage: 50Gi

---
# ── Mount PVC vào Pod ──
containers:
- name: postgres
  volumeMounts:
  - name:      data
    mountPath: /var/lib/postgresql/data
volumes:
- name: data
  persistentVolumeClaim:
    claimName: postgres-data

# ── Access Modes ──
# ReadWriteOnce (RWO)  — 1 node đọc/ghi (block storage như EBS)
# ReadOnlyMany  (ROX)  — nhiều nodes đọc
# ReadWriteMany (RWX)  — nhiều nodes đọc/ghi (NFS, EFS, CephFS)
# ReadWriteOncePod (RWOP) — chỉ 1 pod đọc/ghi (K8s 1.22+)

STATEFULSET

StatefulSet — Stateful Apps (Databases)

Stable network identity, ordered deployment, persistent storage per pod

PostgreSQL với StatefulSet
statefulset.yaml — PostgreSQL production setupStatefulSet
apiVersion: apps/v1
kind: StatefulSet
metadata:
  name:      postgres
  namespace: production
spec:
  serviceName: postgres            # headless service tên
  replicas:    3                    # primary + 2 replicas
  selector:
    matchLabels:
      app: postgres
  updateStrategy:
    type: RollingUpdate
    rollingUpdate:
      partition: 0                 # chỉ update pods có ordinal >= 0
  podManagementPolicy: OrderedReady  # Parallel hoặc OrderedReady
  template:
    metadata:
      labels:
        app: postgres
    spec:
      initContainers:               # chạy trước main container
      - name:    init-permission
        image:   busybox:1.36
        command: ["sh", "-c", "chown -R 999:999 /var/lib/postgresql/data"]
        volumeMounts:
        - name: data
          mountPath: /var/lib/postgresql/data
      containers:
      - name:  postgres
        image: postgres:16-alpine
        env:
        - name:  POSTGRES_DB
          value: shopdb
        - name:  POSTGRES_USER
          value: admin
        - name: POSTGRES_PASSWORD
          valueFrom:
            secretKeyRef:
              name: postgres-secrets
              key:  password
        # Biết pod number thông qua downward API
        - name: POD_NAME
          valueFrom:
            fieldRef:
              fieldPath: metadata.name
        ports:
        - containerPort: 5432
        volumeMounts:
        - name:      data
          mountPath: /var/lib/postgresql/data
        resources:
          requests: {memory: 1Gi, cpu: 500m}
          limits:   {memory: 2Gi, cpu: 2000m}
  volumeClaimTemplates:           # mỗi pod tạo PVC riêng!
  - metadata:
      name: data
    spec:
      accessModes:      [ReadWriteOnce]
      storageClassName: fast-ssd
      resources:
        requests:
          storage: 50Gi
# Pods được tạo: postgres-0, postgres-1, postgres-2
# DNS:         postgres-0.postgres.production.svc.cluster.local

# ── DaemonSet — chạy 1 pod trên EVERY node ──
apiVersion: apps/v1
kind: DaemonSet
metadata:
  name: fluentd
spec:
  selector:
    matchLabels:
      app: fluentd
  template:
    metadata:
      labels:
        app: fluentd
    spec:
      tolerations:                   # chạy cả trên control plane nodes
      - key:      node-role.kubernetes.io/control-plane
        effect:   NoSchedule
      containers:
      - name:  fluentd
        image: fluent/fluentd-kubernetes-daemonset:v1
        volumeMounts:
        - name:      varlog
          mountPath: /var/log
      volumes:
      - name: varlog
        hostPath:
          path: /var/log

---
# ── CronJob — backup database hàng ngày ──
apiVersion: batch/v1
kind: CronJob
metadata:
  name:      postgres-backup
  namespace: production
spec:
  schedule:          "0 2 * * *"       # 2AM hàng ngày
  timeZone:          "Asia/Ho_Chi_Minh"
  successfulJobsHistoryLimit: 3
  failedJobsHistoryLimit:     3
  concurrencyPolicy:  Forbid            # Forbid | Allow | Replace
  jobTemplate:
    spec:
      backoffLimit:    3               # retry 3 lần nếu fail
      activeDeadlineSeconds: 3600   # timeout 1 giờ
      template:
        spec:
          restartPolicy: OnFailure  # Job phải là OnFailure hoặc Never
          containers:
          - name:    backup
            image:   postgres:16-alpine
            command: ["sh", "-c"]
            args:
            - |
              pg_dump -h postgres -U admin shopdb | gzip | \
              aws s3 cp - s3://company-backups/postgres/$(date +%Y%m%d).sql.gz

Ingress & IngressClass — HTTP Routing

Ingress với TLS và multiple services
ingress.yaml — NGINX Ingress với TLS + routingIngress
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name:      api-ingress
  namespace: production
  annotations:
    kubernetes.io/ingress.class:               nginx
    nginx.ingress.kubernetes.io/ssl-redirect:  "true"
    nginx.ingress.kubernetes.io/proxy-body-size:50m
    nginx.ingress.kubernetes.io/rate-limit:    "100"
    cert-manager.io/cluster-issuer:           letsencrypt-prod
spec:
  ingressClassName: nginx
  tls:
  - hosts:
    - api.company.com
    secretName: api-tls-secret          # cert-manager tự tạo
  rules:
  - host: api.company.com
    http:
      paths:
      - path:     /api/orders
        pathType: Prefix
        backend:
          service:
            name: order-service
            port: {number: 80}
      - path:     /api/payments
        pathType: Prefix
        backend:
          service:
            name: payment-service
            port: {number: 80}
      - path:     /
        pathType: Prefix
        backend:
          service:
            name: api-gateway
            port: {number: 80}

# Flow: Internet → LoadBalancer → NGINX Ingress Controller → Service → Pods

AUTOSCALING

HPA & VPA — Tự động Scale

Horizontal Pod Autoscaler scale replicas · Vertical scale resources

HPA với CPU, Memory và custom metrics
hpa.yaml — CPU + Memory + custom metricsHPA
apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name:      order-service-hpa
  namespace: production
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind:       Deployment
    name:       order-service
  minReplicas: 3
  maxReplicas: 20
  metrics:
  - type: Resource
    resource:
      name: cpu
      target:
        type:               Utilization
        averageUtilization: 70         # scale khi CPU > 70%
  - type: Resource
    resource:
      name: memory
      target:
        type:          AverageValue
        averageValue:  400Mi
  - type: Pods                          # custom Prometheus metric
    pods:
      metric:
        name: http_requests_per_second
      target:
        type:         AverageValue
        averageValue: 1000
  behavior:
    scaleUp:
      stabilizationWindowSeconds: 60    # chờ 60s trước khi scale up
      policies:
      - type:          Pods
        value:         4
        periodSeconds: 60              # thêm tối đa 4 pods mỗi phút
    scaleDown:
      stabilizationWindowSeconds: 300   # chờ 5 phút trước khi scale down
      policies:
      - type:          Percent
        value:         10
        periodSeconds: 60              # giảm tối đa 10% mỗi phút

Resource Requests & Limits — Tối ưu tài nguyên

QoS Classes, CPU throttling, OOM killer — hiểu sâu về resource management

QoS Classes và ảnh hưởng khi node thiếu tài nguyên
Resource QoS — Guaranteed, Burstable, BestEffortResources
# ── Guaranteed QoS (requests == limits) ──
# → Ưu tiên cao nhất, không bị evict trừ khi hết tài nguyên tuyệt đối
resources:
  requests: {memory: "512Mi", cpu: "500m"}
  limits:   {memory: "512Mi", cpu: "500m"}

# ── Burstable QoS (requests < limits hoặc chỉ có 1) ──
# → Bị evict sau BestEffort, có thể burst lên limits khi cần
resources:
  requests: {memory: "256Mi", cpu: "250m"}
  limits:   {memory: "1Gi",   cpu: "2000m"}

# ── BestEffort QoS (không có requests/limits) ──
# → Bị evict đầu tiên. KHÔNG dùng cho production!

# ── CPU units: 1 = 1 vCPU/core ──
# 1000m = 1 CPU  |  500m = 0.5 CPU  |  100m = 0.1 CPU

# ── Memory units ──
# 1Gi = 1 GiB (binary) = 1073741824 bytes
# 1G  = 1 GB  (decimal) = 1000000000 bytes
# 512Mi = 512 MiB

# ── Thực tế cho Spring Boot ──
resources:
  requests:
    memory: "512Mi"     # JVM heap ~256-512MB
    cpu:    "250m"      # idle JVM dùng ~100-200m
  limits:
    memory: "1Gi"       # allow burst cho GC
    cpu:    "1000m"     # peak load

# ── JVM flags quan trọng khi chạy trong container ──
env:
- name: JAVA_OPTS
  value: "-XX:+UseContainerSupport -XX:MaxRAMPercentage=75.0 -XX:+ExitOnOutOfMemoryError"
# UseContainerSupport: JVM đọc limits từ cgroups thay vì host RAM
# MaxRAMPercentage=75: dùng 75% của memory limit
# ExitOnOutOfMemoryError: exit để K8s restart, không treo zombie
PROBES

Probes — Health Checks

liveness, readiness, startup probes — tránh mọi downtime

3 loại probe và khi nào dùng
probes — liveness, readiness, startupProbes
# livenessProbe:  App có alive không? → restart container nếu fail
# readinessProbe: App có sẵn sàng nhận traffic không? → remove từ LB nếu fail
# startupProbe:   App đã khởi động xong chưa? → disable liveness trong startup

containers:
- name: order-service

  # startupProbe — cho app khởi động chậm (Spring Boot ~60s)
  startupProbe:
    httpGet:
      path: /actuator/health
      port: 8080
    failureThreshold:    30         # 30 * 10s = 5 phút max startup time
    periodSeconds:       10

  # livenessProbe — app có còn healthy không?
  livenessProbe:
    httpGet:
      path: /actuator/health/liveness
      port: 8080
    initialDelaySeconds: 0           # 0 vì startupProbe đã handle startup
    periodSeconds:       10
    timeoutSeconds:      5
    failureThreshold:    3           # fail 3 lần → restart
    successThreshold:    1

  # readinessProbe — app có ready nhận traffic không?
  readinessProbe:
    httpGet:
      path: /actuator/health/readiness
      port: 8080
    periodSeconds:    5
    failureThreshold: 3             # fail 3 lần → remove từ Service endpoints
    successThreshold: 1

# ── Spring Boot 3.x Actuator config ──
# application.yml:
# management.health.livenessState.enabled: true
# management.health.readinessState.enabled: true
# Readiness fail khi: DB unavailable, Kafka unavailable → không nhận request
# Liveness fail khi: deadlock, memory leak → restart

# ── Probe types ──
# httpGet:  HTTP GET, success nếu 200-399
# exec:     chạy command, success nếu exit 0
# tcpSocket:TCP connect, success nếu connect được
# grpc:     gRPC health check protocol
PDB

Pod Disruption Budget — Đảm bảo Availability

Ngăn K8s drain quá nhiều pods cùng lúc trong maintenance

PDB đảm bảo minimum availability
poddisruptionbudget.yamlPDB
apiVersion: policy/v1
kind: PodDisruptionBudget
metadata:
  name:      order-service-pdb
  namespace: production
spec:
  minAvailable: 2                    # luôn có ít nhất 2 pods available
  # hoặc: maxUnavailable: 1          # tối đa 1 pod unavailable
  # hoặc: minAvailable: "80%"         # tối thiểu 80% pods
  selector:
    matchLabels:
      app: order-service

# Khi kubectl drain node hoặc cluster upgrade:
# → K8s sẽ không drain node nếu PDB bị vi phạm
# → Đảm bảo always có ít nhất 2 replicas serving traffic

# Xem PDB status
$ kubectl get pdb -n production
NAME                 MIN AVAILABLE  MAX UNAVAILABLE  ALLOWED DISRUPTIONS  AGE
order-service-pdb    2              N/A              1                    5d

RBAC & ServiceAccount — Bảo mật K8s

Role, ClusterRole, RoleBinding, ServiceAccount — phân quyền chi tiết

RBAC cho CI/CD và microservices
rbac.yaml — ServiceAccount + Role + RoleBindingRBAC
# ── ServiceAccount cho App ──
apiVersion: v1
kind: ServiceAccount
metadata:
  name:      order-service-sa
  namespace: production
  annotations:
    eks.amazonaws.com/role-arn: arn:aws:iam::123456:role/order-service-role
automountServiceAccountToken: false  # tắt auto-mount nếu không cần K8s API

---
# ── Role — quyền trong namespace ──
apiVersion: rbac.authorization.k8s.io/v1
kind: Role
metadata:
  name:      order-service-role
  namespace: production
rules:
- apiGroups: [""]                    # "" = core API group
  resources: [configmaps, secrets]
  verbs:     [get, list, watch]
- apiGroups: [""]
  resources: [pods]
  verbs:     [get, list]

---
# ── RoleBinding — gán Role cho ServiceAccount ──
apiVersion: rbac.authorization.k8s.io/v1
kind: RoleBinding
metadata:
  name:      order-service-rolebinding
  namespace: production
subjects:
- kind:      ServiceAccount
  name:      order-service-sa
  namespace: production
roleRef:
  kind:     Role
  name:     order-service-role
  apiGroup: rbac.authorization.k8s.io

---
# ── ClusterRole cho CI/CD (toàn cluster) ──
apiVersion: rbac.authorization.k8s.io/v1
kind: ClusterRole
metadata:
  name: ci-deploy-role
rules:
- apiGroups: [apps]
  resources: [deployments, statefulsets]
  verbs:     [get, list, create, update, patch]
- apiGroups: [""]
  resources: [services, configmaps, secrets]
  verbs:     [get, list, create, update, patch]

  repository: registry.company.com/order-service

  pullPolicy: IfNotPresent

replicaCount: 3

service:
  type: ClusterIP
  port: 80

resources:
  requests: {cpu: 250m, memory: 256Mi}
  limits:   {cpu: 1000m, memory: 512Mi}

autoscaling:
  enabled:     true
  minReplicas: 3
  maxReplicas: 20
  targetCPUUtilizationPercentage: 70

ingress:
  enabled:     true
  className:   nginx
  host:        api.company.com

apiVersion: apps/v1

  name: {{ include "order-service.fullname" . }}

apiVersion: apps/v1
kind: Deployment
metadata:
  name:      order-service

          periodSeconds: 5

          allowPrivilegeEscalation: false

Lộ trình học K8s — từ zero đến production
1

Setup local cluster — minikube hoặc kind

Cài minikube start hoặc kind create cluster. Làm quen kubectl với namespace default. Apply Pod và Deployment đầu tiên.

2

Nắm Core Objects: Pod → Deployment → Service

Tạo Deployment với 3 replicas. Expose qua ClusterIP Service. Test DNS resolution bên trong cluster. Hiểu label selectors.

3

ConfigMap, Secret, Volumes

Đưa toàn bộ config ra ConfigMap. Secret cho passwords. Mount ConfigMap làm file application.yml. PVC cho stateful apps.

4

Ingress, HPA, Probes, Resources

Setup NGINX Ingress với TLS. Thêm liveness/readiness/startup probes. Set resource requests và limits. Enable HPA với CPU trigger.

5

Helm Charts và GitOps

Đóng gói toàn bộ manifest thành Helm Chart. Tạo values-dev.yaml, values-prod.yaml riêng. Tích hợp helm upgrade --install vào GitLab CI.

6

RBAC, Network Policies, Security hardening

ServiceAccount per deployment. RBAC với least privilege. Network Policy restrict traffic. SecurityContext non-root, readOnlyRootFilesystem. PDB cho zero-downtime maintenance.
