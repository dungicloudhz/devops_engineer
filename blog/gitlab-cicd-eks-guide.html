Khái niệm · 02

💡 Concepts cơ bản

Hiểu rõ các thành phần trước khi bắt đầu giúp bạn debug và tùy chỉnh pipeline nhanh hơn nhiều.

🦊
GitLab CI/CD
File .gitlab-ci.yml định nghĩa toàn bộ pipeline. GitLab Runner nhận job, thực thi trong Docker container hoặc Kubernetes pod, rồi báo cáo kết quả về GitLab.
☸️
Amazon EKS
Managed Kubernetes service của AWS. EKS quản lý control plane (API server, etcd), bạn chỉ cần lo worker nodes qua Node Groups hoặc Fargate.
📦
Amazon ECR
Elastic Container Registry — private Docker registry của AWS. GitLab CI build image, push lên ECR, rồi EKS pull về khi deploy.
⛵
Helm Charts
Package manager cho Kubernetes. Helm cho phép template hóa manifest YAML, quản lý version, rollback dễ dàng với helm rollback.
🔑
IRSA (IAM Roles for SA)
Cơ chế best practice để cấp quyền AWS cho pod trong EKS mà không cần access key cứng. GitLab Runner pod dùng IAM role để push ECR, deploy EKS.
🌿
GitLab Environments
Mỗi nhánh dev, staging, main map tới một environment. Cho phép track deployment history, rollback UI, và protected deployments.

☸️ Cài đặt EKS Cluster

Tạo EKS cluster bằng eksctl — công cụ CLI chính thức, đơn giản nhất để bootstrap EKS.

📄 Tạo cluster config file

cluster.yaml
# cluster.yaml
apiVersion: eksctl.io/v1alpha5
kind: ClusterConfig

metadata:
  name: my-app-cluster
  region: ap-southeast-1    # Singapore
  version: "1.30"

iam:
  withOIDC: true               # Bắt buộc cho IRSA

managedNodeGroups:
  - name: workers
    instanceType: t3.medium
    minSize: 2
    maxSize: 6
    desiredCapacity: 3
    amiFamily: AmazonLinux2
    privateNetworking: true  # Nodes ở private subnet
    labels:
      role: worker
    tags:
      Environment: production
      Team: platform

addons:
  - name: vpc-cni
  - name: coredns
  - name: kube-proxy
  - name: aws-ebs-csi-driver

cloudWatch:
  clusterLogging:
    enableTypes: ["api", "audit", "authenticator"]
1
Tạo cluster
Chạy lệnh bên dưới — quá trình mất khoảng 15-20 phút để AWS provision đầy đủ.
bash
# Tạo cluster từ config file
eksctl create cluster -f cluster.yaml

# Xác nhận cluster đã sẵn sàng
kubectl get nodes
kubectl get pods -A
2
Cài đặt AWS Load Balancer Controller
Cần thiết để expose service ra ngoài qua ALB/NLB.
bash
# Thêm Helm repo của EKS
helm repo add eks https://aws.github.io/eks-charts
helm repo update

# Cài AWS Load Balancer Controller
helm install aws-load-balancer-controller eks/aws-load-balancer-controller \
  -n kube-system \
  --set clusterName=my-app-cluster \
  --set serviceAccount.create=false \
  --set serviceAccount.name=aws-load-balancer-controller
Thiết lập · 05

🏃 Cài đặt GitLab Runner trên EKS

Chạy GitLab Runner ngay trong EKS cluster bằng Helm chart chính thức. Runner sẽ tạo pod riêng cho mỗi CI job.

✅ Tại sao deploy Runner vào EKS?
Runner chạy trong cluster → có thể dùng IRSA để lấy AWS credentials tự động, không cần access key cứng. Mỗi job chạy trong pod isolated, sau khi xong pod tự xóa.

🔑 Lấy Registration Token từ GitLab

Vào GitLab → Project → Settings → CI/CD → Runners → New project runner. Chọn tag (ví dụ: eks), copy Registration Token.

bash · Cài GitLab Runner qua Helm
# Thêm GitLab Helm repo
helm repo add gitlab https://charts.gitlab.io
helm repo update

# Tạo namespace cho runner
kubectl create namespace gitlab-runner

# Cài đặt runner
helm install gitlab-runner gitlab/gitlab-runner \
  --namespace gitlab-runner \
  --set gitlabUrl=https://gitlab.com \
  --set runnerRegistrationToken=YOUR_TOKEN_HERE \
  --set runners.tags="eks,docker,kubernetes" \
  --set runners.privileged=true \
  --set rbac.create=true \
  --set runners.executor=kubernetes

⚙️ Tuỳ chỉnh Runner với values.yaml

runner-values.yaml
gitlabUrl: https://gitlab.com

runners:
  executor: kubernetes
  tags: "eks,aws"
  privileged: true        # Cần cho Docker-in-Docker

  kubernetes:
    namespace: gitlab-runner
    image: alpine:3.18
    cpuLimit: "2"
    memoryLimit: 2Gi
    cpuRequests: "500m"
    memoryRequests: 512Mi
    serviceAccountName: gitlab-runner  # SA có IRSA

    volumes:
      - name: docker-sock
        hostPath:
          path: /var/run/docker.sock

rbac:
  create: true
  clusterWideAccess: false

metrics:
  enabled: true   # Expose Prometheus metrics
bash
# Cài với file values
helm upgrade --install gitlab-runner gitlab/gitlab-runner \
  --namespace gitlab-runner \
  -f runner-values.yaml

# Kiểm tra runner pod đã Running chưa
kubectl get pods -n gitlab-runner
Thiết lập · 06

🔑 IAM & ECR — Xác thực không cần Secret Key

Sử dụng IRSA (IAM Roles for Service Accounts) thay vì hardcode AWS_ACCESS_KEY_ID. An toàn hơn và không cần rotate key thủ công.

⚠️ Đừng hardcode AWS credentials
Không bao giờ đặt AWS_ACCESS_KEY_ID và AWS_SECRET_ACCESS_KEY trực tiếp trong .gitlab-ci.yml. Dùng IRSA hoặc GitLab CI Variables (masked + protected).

1️⃣ Tạo ECR Repository

bash · AWS CLI
# Tạo ECR repo cho app
aws ecr create-repository \
  --repository-name my-app \
  --region ap-southeast-1 \
  --image-scanning-configuration scanOnPush=true \
  --encryption-configuration encryptionType=AES256

# Lấy ECR URL
aws ecr describe-repositories \
  --repository-names my-app \
  --query 'repositories[0].repositoryUri' \
  --output text

2️⃣ Tạo IAM Role cho GitLab Runner (IRSA)

bash · eksctl
# Tạo IAM service account với IRSA
eksctl create iamserviceaccount \
  --name gitlab-runner \
  --namespace gitlab-runner \
  --cluster my-app-cluster \
  --attach-policy-arn arn:aws:iam::aws:policy/AmazonEC2ContainerRegistryPowerUser \
  --attach-policy-arn arn:aws:iam::aws:policy/AmazonEKSClusterPolicy \
  --approve \
  --override-existing-serviceaccounts

3️⃣ Thiết lập GitLab CI Variables

Vào GitLab → Project → Settings → CI/CD → Variables và thêm các biến sau:

VariableGiá trị mẫuMaskedProtected
AWS_REGIONap-southeast-1——
AWS_ACCOUNT_ID123456789012✅—
ECR_REGISTRY123456789012.dkr.ecr.ap-southeast-1.amazonaws.com——
ECR_REPOSITORYmy-app——
EKS_CLUSTER_NAMEmy-app-cluster——
KUBE_NAMESPACE_PRODproduction—✅
✅ Tip: Dùng GitLab OIDC với AWS
GitLab 15.7+ hỗ trợ native OIDC federation với AWS. Cho phép CI job lấy AWS credentials tạm thời qua id_token mà không cần access key nào. Xem phần "Nâng cao" phía dưới để biết thêm.
Pipeline · 07

⚙️ Pipeline .gitlab-ci.yml cơ bản

File .gitlab-ci.yml đặt ở root của repo. Đây là bộ khung đầy đủ cho một pipeline production-ready.

.gitlab-ci.yml · Full pipeline
# ─────────────────────────────────────────
# GitLab CI/CD Pipeline for EKS Deployment
# ─────────────────────────────────────────

image: docker:24-dind

variables:
  DOCKER_TLS_CERTDIR: "/certs"
  DOCKER_DRIVER: overlay2
  IMAGE_TAG: $CI_REGISTRY_IMAGE:$CI_COMMIT_SHA
  HELM_RELEASE_NAME: my-app
  HELM_CHART_PATH: ./charts/my-app

stages:
  - test
  - build
  - scan
  - deploy-dev
  - deploy-staging
  - deploy-prod

# ── Shared config ──────────────────────────
.aws-auth: &aws-auth
  before_script:
    - aws ecr get-login-password --region $AWS_REGION |
      docker login --username AWS --password-stdin $ECR_REGISTRY
    - aws eks update-kubeconfig
        --name $EKS_CLUSTER_NAME
        --region $AWS_REGION

# ── Stage: TEST ────────────────────────────
unit-test:
  stage: test
  image: node:20-alpine
  tags: [eks]
  cache:
    key: $CI_COMMIT_REF_SLUG
    paths: [node_modules/]
  script:
    - npm ci
    - npm run lint
    - npm test -- --coverage
  coverage: '/Lines\s*:\s*(\d+\.?\d*)%/'
  artifacts:
    reports:
      coverage_report:
        coverage_format: cobertura
        path: coverage/cobertura-coverage.xml

# ── Stage: BUILD ───────────────────────────
build-image:
  stage: build
  tags: [eks]
  <<: *aws-auth
  services: [docker:24-dind]
  script:
    - export FULL_IMAGE=$ECR_REGISTRY/$ECR_REPOSITORY:$CI_COMMIT_SHA
    - docker build
        --cache-from $ECR_REGISTRY/$ECR_REPOSITORY:latest
        --tag $FULL_IMAGE
        --tag $ECR_REGISTRY/$ECR_REPOSITORY:latest
        --build-arg BUILD_DATE=$(date -u +"%Y-%m-%dT%H:%M:%SZ")
        --build-arg VCS_REF=$CI_COMMIT_SHA .
    - docker push $FULL_IMAGE
    - docker push $ECR_REGISTRY/$ECR_REPOSITORY:latest
  only: [main, develop, staging]

# ── Stage: SCAN ────────────────────────────
trivy-scan:
  stage: scan
  image: aquasec/trivy:latest
  tags: [eks]
  script:
    - trivy image
        --exit-code 1
        --severity HIGH,CRITICAL
        --no-progress
        $ECR_REGISTRY/$ECR_REPOSITORY:$CI_COMMIT_SHA
  allow_failure: false

# ── Stage: DEPLOY DEV ──────────────────────
deploy-dev:
  stage: deploy-dev
  image: alpine/helm:3.14.0
  tags: [eks]
  <<: *aws-auth
  environment:
    name: dev
    url: https://dev.myapp.com
  script:
    - helm upgrade --install $HELM_RELEASE_NAME $HELM_CHART_PATH
        --namespace dev --create-namespace
        --set image.repository=$ECR_REGISTRY/$ECR_REPOSITORY
        --set image.tag=$CI_COMMIT_SHA
        --set ingress.host=dev.myapp.com
        --values charts/values-dev.yaml
        --wait --timeout 5m
  only: [develop]
ℹ️ YAML Anchors
Dòng &aws-auth và <<: *aws-auth là YAML anchor — giúp tái sử dụng before_script không cần copy-paste. Rất hữu ích cho multi-stage pipeline.
Pipeline · 08

🔄 Giải thích từng Stage

🧪 Test — Unit test & Lint
Chạy mọi commit. Dùng cache node_modules để tăng tốc. Coverage report upload lên GitLab để hiển thị trên MR. Nếu fail → pipeline dừng ngay, không build.
🐳 Build — Docker image & push ECR
Dùng --cache-from latest để tận dụng Docker layer cache → giảm build time 50-70%. Tag image bằng $CI_COMMIT_SHA để traceable 100%.
🔍 Scan — Trivy vulnerability scan
Scan image trước khi deploy. Exit code 1 nếu có lỗ hổng HIGH/CRITICAL. Có thể thêm --ignore-unfixed để chỉ report lỗ đã có fix.
⛵ Deploy — Helm upgrade/install
helm upgrade --install là idempotent — an toàn gọi nhiều lần. Flag --wait đảm bảo pipeline chỉ pass khi pods thực sự Running, không phải chỉ khi manifest apply xong.
✅ Verify — Smoke test sau deploy
Gọi health endpoint của app, xác nhận đúng version vừa deploy. Nếu fail → trigger rollback tự động.

🎛️ Job control nâng cao

.gitlab-ci.yml · Rules thay vì only/except
deploy-prod:
  stage: deploy-prod
  rules:
    # Chỉ chạy trên branch main
    - if: '$CI_COMMIT_BRANCH == "main"'
      when: manual       # Cần approval thủ công
    # Không chạy nếu commit message có [skip prod]
    - if: '$CI_COMMIT_MESSAGE =~ /\[skip prod\]/'
      when: never
  environment:
    name: production
    url: https://myapp.com
    deployment_tier: production
  allow_failure: false

📄 deployment.yaml template

templates/deployment.yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: {{ include "my-app.fullname" . }}
  labels: {{- include "my-app.labels" . | nindent 4 }}
spec:
  replicas: {{ .Values.replicaCount }}
  selector:
    matchLabels: {{- include "my-app.selectorLabels" . | nindent 6 }}

🔀 Pipeline đầy đủ cho 3 môi trường

.gitlab-ci.yml · Multi-env
# Template job deploy tái sử dụng
.deploy-template: &deploy-template
  image: dtzar/helm-kubectl:3.14
  tags: [eks]
  before_script:
    - aws eks update-kubeconfig --name $EKS_CLUSTER_NAME --region $AWS_REGION
    - aws ecr get-login-password | docker login --username AWS --password-stdin $ECR_REGISTRY
  script:
    - helm upgrade --install my-app ./charts/my-app
        --namespace $DEPLOY_NAMESPACE
        --create-namespace
        --set image.tag=$CI_COMMIT_SHA
        --set image.repository=$ECR_REGISTRY/$ECR_REPOSITORY
        -f ./charts/my-app/values-$DEPLOY_ENV.yaml
        --wait --timeout 10m
        --atomic    # Auto rollback nếu deploy fail

deploy-dev:
  <<: *deploy-template
  stage: deploy-dev
  variables:
    DEPLOY_ENV: dev
    DEPLOY_NAMESPACE: dev
  environment: { name: dev, url: https://dev.myapp.com }
  rules:
    - if: '$CI_COMMIT_BRANCH == "develop"'

deploy-staging:
  <<: *deploy-template
  stage: deploy-staging
  variables:
    DEPLOY_ENV: staging

    - if: '$CI_COMMIT_BRANCH == "staging"'

deploy-prod:
  <<: *deploy-template
  stage: deploy-prod
  variables:
    DEPLOY_ENV: prod

🔐 Secrets & Configuration Management

Không bao giờ lưu secrets trong code. Dùng AWS Secrets Manager + External Secrets Operator để inject secret vào Kubernetes tự động.

🔐
AWS Secrets Manager
Lưu trữ secrets tập trung, có versioning, auto-rotation. Tích hợp tốt với EKS qua External Secrets Operator.
⚙️
External Secrets Operator
Sync secret từ AWS Secrets Manager / SSM Parameter Store vào Kubernetes Secret tự động. Không cần copy secret thủ công.
🗝️
GitLab CI Variables
Non-sensitive config (URLs, region, cluster name) đặt trong GitLab Variables. Sensitive data dùng AWS Secrets Manager.

📦 Cài External Secrets Operator

bash
helm repo add external-secrets https://charts.external-secrets.io
helm install external-secrets external-secrets/external-secrets \
  -n external-secrets --create-namespace

📄 ExternalSecret manifest

external-secret.yaml
apiVersion: external-secrets.io/v1beta1
kind: ExternalSecret
metadata:
  name: my-app-secrets
  namespace: production
spec:
  refreshInterval: 1h          # Sync mỗi 1 giờ
  secretStoreRef:
    name: aws-secretsmanager
    kind: ClusterSecretStore
  target:
    name: my-app-secrets         # Kubernetes Secret tạo ra
    creationPolicy: Owner
  data:
    - secretKey: DATABASE_URL
      remoteRef:
        key: production/my-app/database
        property: url
    - secretKey: REDIS_PASSWORD
      remoteRef:
        key: production/my-app/redis
        property: password

🔓 GitLab OIDC với AWS (không cần access key)

.gitlab-ci.yml · OIDC Auth
assume-role-job:
  id_tokens:
    GITLAB_OIDC_TOKEN:
      aud: https://gitlab.com
  script:
    - export $(printf "AWS_ACCESS_KEY_ID=%s AWS_SECRET_ACCESS_KEY=%s AWS_SESSION_TOKEN=%s"
      $(aws sts assume-role-with-web-identity
        --role-arn arn:aws:iam::$AWS_ACCOUNT_ID:role/GitLabCIRole
        --role-session-name GitLabCI
        --web-identity-token $GITLAB_OIDC_TOKEN
        --query 'Credentials.[AccessKeyId,SecretAccessKey,SessionToken]'
        --output text))
    - aws sts get-caller-identity   # Xác nhận đã auth thành công

↩️ Rollback & Canary Deployment

⚡ Rollback nhanh với Helm

bash · Rollback commands
# Xem lịch sử release
helm history my-app -n production

# Rollback về revision trước
helm rollback my-app -n production

# Rollback về revision cụ thể
helm rollback my-app 3 -n production --wait

🐦 Rollback tự động trong pipeline

.gitlab-ci.yml · Auto rollback
deploy-prod:
  script:
    # --atomic: nếu deploy fail, Helm tự rollback
    - helm upgrade --install my-app ./charts/my-app
        --namespace production
        --set image.tag=$CI_COMMIT_SHA
        --values ./charts/my-app/values-prod.yaml
        --wait --timeout 10m
        --atomic
    # Smoke test sau deploy
    - export APP_URL=https://myapp.com
    - curl -f $APP_URL/health || (helm rollback my-app -n production && exit 1)

rollback-prod:
  stage: deploy-prod
  when: manual       # Trigger thủ công khi cần
  script:
    - helm rollback my-app -n production --wait
  environment:
    name: production
    action: stop
  rules:
    - if: '$CI_COMMIT_BRANCH == "main"'

🕊️ Canary Deployment với Argo Rollouts

ℹ️ Argo Rollouts
Thay Deployment bằng Rollout của Argo để có canary, blue/green native trong Kubernetes. Kết hợp với GitLab pipeline để progressive delivery.
rollout.yaml · Canary strategy
apiVersion: argoproj.io/v1alpha1
kind: Rollout
metadata:
  name: my-app
spec:
  strategy:
    canary:
      steps:
        - setWeight: 10       # 10% traffic → canary

Best Practices · 14

✨ Best Practices & Tips

  • Dùng --atomic trong helm upgrade — Tự động rollback nếu deployment không healthy trong timeout. Tránh trạng thái stuck giữa chừng.
  • Tag image bằng $CI_COMMIT_SHA — Luôn traceable, biết chính xác code nào đang chạy. Không bao giờ dùng latest trên production.
  • Không dùng :latest tag cho base image — Pin version cụ thể (node:20.11-alpine) để build reproducible, tránh surprise.
  • Cache node_modules / pip packages — Dùng GitLab CI cache key theo $CI_COMMIT_REF_SLUG kết hợp lock file hash để tăng tốc 3-5x.
  • Dùng Multi-stage Dockerfile — Builder stage riêng, final image chỉ có runtime artifacts. Giảm image size 70-80%, bề mặt tấn công nhỏ hơn.
  • Set resource requests và limits — Không để Kubernetes tự đoán. Requests đảm bảo scheduling, limits tránh noisy neighbor.
  • Luôn có readinessProbe và livenessProbe — Kubernetes cần biết khi nào pod ready nhận traffic và khi nào cần restart.
  • Namespace riêng cho mỗi môi trường — Cách ly dev/staging/prod bằng namespace. Kết hợp với RBAC để phân quyền rõ ràng.
  • Protected branches + Protected environments — Chỉ Maintainer mới được deploy production. Kết hợp với required approvals.
  • Lưu pipeline artifacts có expiry — Test reports, coverage files: expire sau 1 tuần. Image artifacts không lưu trong GitLab, dùng ECR.
❌ Anti-patterns cần tránh
Không hardcode credentials trong .gitlab-ci.yml dù là secrets hay không. Không chạy production deploy tự động không cần approval. Không bỏ qua security scan vì "sẽ sửa sau". Không dùng privileged: true cho runner production nếu không cần thiết.
