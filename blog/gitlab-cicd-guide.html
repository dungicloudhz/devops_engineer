GitLab vs GitHub — Khi nào chọn GitLab?
🔄
Built-in CI/CD
CI/CD native — không cần Jenkins, GitHub Actions. Cấu hình bằng .gitlab-ci.yml ngay trong repo.
🛡️
Security Scanning
SAST, DAST, Container Scanning, Dependency Scanning built-in. Tích hợp vào pipeline tự động.
🏠
Self-hosted
Cài đặt trên server riêng (GitLab CE/EE). Phù hợp với enterprise, ngân hàng, fintech cần data sovereignty.
📦
Container Registry
Registry Docker images tích hợp sẵn. Build và push image ngay trong pipeline không cần DockerHub.

BRANCHES

Branch Strategy — GitFlow & Trunk-based

Hai chiến lược phổ biến — chọn đúng cho team size và release cadence

GitFlow vs Trunk-based Development
🌿 GitFlow
main → production code
develop → integration branch
feature/* → new features
release/* → release prep
hotfix/* → production fixes
✅ Rõ ràng, phù hợp versioned releases
❌ Nhiều branches, merge phức tạp
🚀 Trunk-based Development
main → luôn deployable
feature/* → short-lived (<1 ngày)
release/ → optional tags
✅ CI/CD nhanh, ít conflict
✅ Feature flags control releases
❌ Cần team discipline cao
🔶 Thực tế
Branch Protection Rules quan trọng
Settings → Repository → Protected Branches: main — Allowed to merge: Maintainers only, No force push. develop — Allowed to merge: Developers+Maintainers via MR.
Push Rules: Commit message format, no secrets in code, require signed commits (enterprise).

.gitlab-ci.yml — full structure annotatedCI Config
# ── Global defaults ──
default:
  image: eclipse-temurin:21-jdk         # Docker image cho tất cả jobs
  tags:                                  # runner tags
    - docker
    - linux
  interruptible: true                   # cancel khi có pipeline mới
  retry:                                 # retry khi runner fail
    max: 2
    when: runner_system_failure

# ── Stages — thứ tự chạy ──
stages:
  - build
  - test
  - package
  - deploy-dev
  - deploy-staging
  - deploy-prod

# ── Global variables ──
variables:
  MAVEN_OPTS: "-Dmaven.repo.local=$CI_PROJECT_DIR/.m2/repository"
  IMAGE_NAME: "$CI_REGISTRY_IMAGE/$CI_PROJECT_NAME"
  IMAGE_TAG:  "$CI_COMMIT_SHORT_SHA"
  APP_NAME:   "order-service"

# ── Job định nghĩa ──
compile:
  stage: build
  script:
    - ./mvnw compile -DskipTests
  artifacts:
    paths:
      - target/
    expire_in: 1 hour

unit-test:
  stage: test
  script:
    - ./mvnw test
  coverage: '/Total.*?([0-9]{1,3})%/'   # extract coverage từ output
  artifacts:
    when: always                        # giữ dù pass hay fail
    reports:
      junit: target/surefire-reports/**/*.xml
      coverage_report:
        coverage_format: cobertura
        path: target/site/jacoco/jacoco.xml

docker-build:
  stage: package
  image: docker:24
  services:
    - docker:24-dind                     # Docker-in-Docker
  variables:
    DOCKER_TLS_CERTDIR: "/certs"
  before_script:
    - docker login -u $CI_REGISTRY_USER -p $CI_REGISTRY_PASSWORD $CI_REGISTRY
  script:
    - docker build -t $IMAGE_NAME:$IMAGE_TAG -t $IMAGE_NAME:latest .
    - docker push $IMAGE_NAME:$IMAGE_TAG
    - docker push $IMAGE_NAME:latest
  rules:
    - if: $CI_COMMIT_BRANCH == $CI_DEFAULT_BRANCH
JOBS

Jobs & Stages — Tất cả keywords quan trọng

before_script, after_script, needs, dependencies, when, allow_failure

Job keywords toàn diện
Job keywords — needs, dependencies, when, parallelJobs
# ── needs — chạy job ngay khi dependency xong (không cần chờ stage) ──
integration-test:
  stage: test
  needs:
    - job: compile
      artifacts: true                  # lấy artifacts từ compile job
  services:
    - name: postgres:16-alpine
      alias: db
    - name: redis:7-alpine
  variables:
    SPRING_DATASOURCE_URL: "jdbc:postgresql://db:5432/testdb"
  script:
    - ./mvnw verify -Pintegration-test

# ── when — điều kiện chạy job ──
deploy-prod:
  stage: deploy-prod
  when: manual                          # phải click tay mới chạy
  allow_failure: false                 # block pipeline nếu fail
  environment:
    name: production
    url: https://app.company.com

# when options:
# on_success (default): chạy khi tất cả jobs trước thành công
# on_failure: chạy chỉ khi có job fail (cleanup, notification)
# always: luôn chạy
# manual: chờ user click
# delayed: chạy sau N giây
# never: không chạy (dùng trong rules)

# ── notify on failure ──
notify-failure:
  stage: .post                         # built-in stage, luôn chạy cuối
  when: on_failure
  image: curlimages/curl
  script:
    - |
      curl -X POST $SLACK_WEBHOOK \
        -H 'Content-type: application/json' \
        --data "{\"text\":\"❌ Pipeline failed: $CI_PROJECT_NAME ($CI_COMMIT_REF_NAME)\"}"

# ── parallel — chạy cùng lúc nhiều instances ──
run-tests:
  stage: test
  parallel: 4                           # chạy 4 jobs song song
  script:
    - ./run-tests.sh --shard $CI_NODE_INDEX/$CI_NODE_TOTAL
  # CI_NODE_INDEX: 1,2,3,4 | CI_NODE_TOTAL: 4

# ── timeout ──
slow-build:
  timeout: 30 minutes                   # cancel sau 30 phút
KeywordMục đíchGiá trị phổ biến
stageXếp job vào stage nàobuild, test, deploy, .pre, .post
imageDocker image để chạy jobeclipse-temurin:21, node:20, python:3.12
tagsChọn runner có tags nàydocker, linux, arm64
needsPhụ thuộc vào job cụ thể (DAG)- job: compile
rulesĐiều kiện chạy/bỏ quaif, changes, exists, when
allow_failurePipeline có fail không khi job failtrue/false
environmentTrack deployment environmentsproduction, staging, dev
resource_groupPrevent concurrent deployproduction, staging

[[runners]]

📌
Executor nào nên dùng?
Shell: Đơn giản nhưng runner bị "ô nhiễm" giữa jobs, cần cleanup thủ công.
Docker: Isolate tốt, phổ biến nhất. Mỗi job một container fresh.
Kubernetes: Scale tự động theo workload, phù hợp production lớn.

RULES

Rules & Conditions — Kiểm soát khi nào job chạy

rules thay thế only/except — linh hoạt hơn nhiều

rules — mọi use case phổ biến
rules — if, changes, exists, whenRules
# ── if — dựa vào biến ──
deploy-main:
  rules:
    - if: $CI_COMMIT_BRANCH == "main"          # chỉ chạy trên main
    - if: $CI_COMMIT_TAG                       # hoặc khi có tag

skip-draft:
  rules:
    - if: $CI_MERGE_REQUEST_TITLE =~ /^Draft:/
      when: never                             # skip Draft MRs
    - when: on_success                         # chạy cho các trường hợp khác

# ── changes — chỉ chạy khi file thay đổi ──
backend-lint:
  rules:
    - if: $CI_PIPELINE_SOURCE == "merge_request_event"
      changes:
        - "src/**/*.java"
        - "pom.xml"

# ── exists — kiểm tra file tồn tại ──
terraform-plan:
  rules:
    - exists:
        - "terraform/**/*.tf"

# ── Kết hợp phức tạp ──
deploy-prod:
  rules:
    - if: $CI_COMMIT_TAG =~ /^v[0-9]+\.[0-9]+\.[0-9]+$/  # chỉ semver tags
      when: manual
      allow_failure: false
    - when: never                               # không chạy trong trường hợp khác

# ── workflow rules — điều kiện chạy TOÀN BỘ pipeline ──
workflow:
  rules:
    - if: $CI_MERGE_REQUEST_IID               # chạy cho MRs
    - if: $CI_COMMIT_BRANCH == $CI_DEFAULT_BRANCH  # chạy cho main
    - if: $CI_COMMIT_TAG                       # chạy cho tags
    - when: never                               # bỏ qua tất cả trường hợp khác
ARTIFACTS

Artifacts & Cache — Tối ưu pipeline speed

Artifacts truyền file giữa jobs, Cache tái dùng dependencies

Artifacts vs Cache — khác nhau quan trọng
📦 Artifacts
Truyền file giữa jobs trong pipeline
Upload lên GitLab server
Download cho jobs cần
Có thể download từ UI
expire_in để auto cleanup
🗃️ Cache
Tái sử dụng giữa các pipelines
Lưu trên runner local hoặc distributed
Dùng cho dependencies (.m2, node_modules)
Không đảm bảo luôn tồn tại
Key xác định cache bucket
Artifacts & Cache cấu hình chi tiếtArtifacts/Cache
# ── Artifacts — giữ kết quả build ──
build-jar:
  stage: build
  script:
    - ./mvnw package -DskipTests
  artifacts:
    name: "$CI_JOB_NAME-$CI_COMMIT_REF_SLUG"  # tên artifact
    paths:
      - target/*.jar
    exclude:
      - target/*-sources.jar              # loại trừ
    expire_in: 1 week
    when: always                            # giữ dù job fail
    reports:                               # các loại reports đặc biệt
      junit:      target/surefire-reports/*.xml
      coverage_report:
        coverage_format: cobertura
        path:            target/site/jacoco/jacoco.xml
      sast:       gl-sast-report.json
      container_scanning: gl-container-scanning-report.json

# ── Cache — tái sử dụng Maven/Gradle dependencies ──
variables:
  MAVEN_OPTS: "-Dmaven.repo.local=$CI_PROJECT_DIR/.m2/repository"

build-with-cache:
  cache:
    key:
      files:
        - pom.xml                           # cache invalidate khi pom.xml thay đổi
      prefix: "maven-$CI_COMMIT_REF_SLUG"
    paths:
      - .m2/repository/
    policy: pull-push                      # pull đầu, push cuối (default)
    # policy: pull   → chỉ dùng cache, không update (fast)
    # policy: push   → chỉ update cache, không dùng

# ── Distributed cache với S3 (trong config.toml) ──
[runners.cache]
  Type   = "s3"
  Shared = true
  [runners.cache.s3]
    ServerAddress = "minio.internal:9000"
    BucketName    = "gitlab-runner-cache"
    Insecure      = false

📐 Khi nào dùng cái nào?
extends vs anchors vs !reference
extends: Tốt nhất cho kế thừa toàn bộ job config. GitLab merge thông minh (deep merge).
YAML Anchors (&/*): Tốt cho reuse script arrays. Nhưng chỉ copy, không merge.
!reference: Tốt nhất khi chỉ muốn lấy một section cụ thể (chỉ variables, chỉ script).

FROM eclipse-temurin:21-jdk-jammy AS builder
WORKDIR /build
COPY .mvn/ .mvn/

RUN ./mvnw dependency:go-offline      # cache dependencies layer
COPY src/ src/

FROM eclipse-temurin:21-jre-jammy    # smaller runtime image
WORKDIR /app
# Security: run as non-root
RUN groupadd -r app && useradd -r -g app app
COPY --from=builder /build/target/*.jar app.jar
RUN chown app:app app.jar
USER app
EXPOSE 8080
HEALTHCHECK --interval=30s --timeout=10s \
  CMD curl -f http://localhost:8080/actuator/health || exit 1
ENTRYPOINT ["java", "-XX:+UseContainerSupport", "-jar", "app.jar"]

docker-build:

        -t $IMAGE_BRANCH
        .

    - docker push $IMAGE_LATEST
    - docker push $IMAGE_BRANCH

  artifacts:
    reports:
      dotenv: build.env                  # biến được pass sang dependent jobs!

Tách pipeline thành templates

Tạo repo ci-templates, chuyển common jobs vào đó. Dùng include: project trong từng project. Một lần fix, tất cả projects được cập nhật.

4

Deploy tự động lên dev, manual lên prod

Dev auto-deploy trên push to main. Staging và Prod là manual gate với when: manual và resource_group. Dùng GitLab Environments để track deployments.

5

Thêm Security Scanning

Include template: Security/SAST.gitlab-ci.yml và Secret-Detection.gitlab-ci.yml. Kết quả hiện trực tiếp trong MR Security tab.

6

Optimize: cache, DAG, parallel

Thêm Maven/Gradle cache, dùng needs thay stage dependencies, enable interruptible: true. Mục tiêu: pipeline dưới 5 phút cho feature branches.
