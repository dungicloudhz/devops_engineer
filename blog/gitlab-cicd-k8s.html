02 — Concepts

🧠 Core Concepts

Trước khi setup, cần hiểu rõ các thành phần chính trong pipeline.

⚙️
.gitlab-ci.yml
PIPELINE CONFIG
File cấu hình pipeline, đặt ở root repo. Định nghĩa stages, jobs, rules, conditions.
🏃
GitLab Runner
JOB EXECUTOR
Agent cài trên server/K8s pod, nhận và thực thi jobs từ GitLab CI. Hỗ trợ nhiều executor types.
🔤
CI/CD Variables
SECRETS & CONFIG
Biến môi trường được inject vào jobs. Có thể set ở Group, Project, hoặc Job level. Masked để bảo vệ secrets.
☸️
Deployment
K8S RESOURCE
K8s resource quản lý replicas của pod. Hỗ trợ rolling update, rollback, và health check tự động.
🔑
K8s Secret
SENSITIVE DATA
Lưu sensitive data như DB password, API keys dưới dạng base64. Mount vào pod qua env vars hoặc volume.
⛵
Helm Release
DEPLOYMENT UNIT
Một lần apply Helm chart với specific values. Tracking history, dễ rollback về version trước.
⚠️ Lưu ý quan trọng
Không bao giờ hardcode secrets trong .gitlab-ci.yml hoặc Helm values. Luôn dùng GitLab CI/CD Variables (Masked + Protected) hoặc AWS Secrets Manager.

04 — GitLab CI

⚙️ .gitlab-ci.yml Cơ bản

File .gitlab-ci.yml là trái tim của pipeline. Đây là cấu trúc đầy đủ cho một Java service.

YAML · .gitlab-ci.yml
# ── Global config ──────────────────────────────
image: maven:3.9-eclipse-temurin-17

variables:
  MAVEN_OPTS: "-Dmaven.repo.local=$CI_PROJECT_DIR/.m2/repository"
  IMAGE_NAME: "$AWS_ACCOUNT_ID.dkr.ecr.$AWS_REGION.amazonaws.com/$CI_PROJECT_NAME"
  IMAGE_TAG: "$CI_COMMIT_SHORT_SHA"
  DOCKER_DRIVER: overlay2

stages:
  - test
  - build
  - publish
  - deploy-dev
  - deploy-prod

# ── Cache Maven deps ──────────────────────────
cache:
  key: "$CI_COMMIT_REF_SLUG-maven"
  paths:
    - .m2/repository
  policy: pull-push
05 — Stages & Jobs

🔄 Stages & Jobs Chi Tiết

🧪 Stage: test

YAML · test stage
unit-test:
  stage: test
  script:
    - mvn test -B --no-transfer-progress
  artifacts:
    when: always
    reports:
      junit:
        - target/surefire-reports/TEST-*.xml
    paths:
      - target/surefire-reports/
    expire_in: 7 days
  coverage: '/Total.*?([0-9]{1,3})%/'
  rules:
    - if: '$CI_PIPELINE_SOURCE == "merge_request_event"'
    - if: '$CI_COMMIT_BRANCH == "main"'

integration-test:
  stage: test
  services:
    - name: postgres:15-alpine
      alias: postgres
  variables:
    POSTGRES_DB: testdb
    POSTGRES_USER: test
    POSTGRES_PASSWORD: test
    SPRING_DATASOURCE_URL: "jdbc:postgresql://postgres:5432/testdb"
  script:
    - mvn verify -P integration-test -B --no-transfer-progress

🔨 Stage: build (Maven + Docker)

YAML · build stage
maven-build:
  stage: build
  script:
    - mvn package -DskipTests -B --no-transfer-progress
  artifacts:
    paths:
      - target/*.jar
    expire_in: 1 hour
  rules:
    - if: '$CI_COMMIT_BRANCH =~ /^(main|develop|release\/.*)$/'

docker-build:
  stage: build
  image: docker:24
  services:
    - docker:24-dind
  needs: [maven-build]
  script:
    - docker build -t $IMAGE_NAME:$IMAGE_TAG -t $IMAGE_NAME:latest .
    - docker save $IMAGE_NAME:$IMAGE_TAG | gzip > image.tar.gz
  artifacts:
    paths:
      - image.tar.gz
    expire_in: 1 hour
✅ Best practice
Dùng needs: thay vì để pipeline chờ toàn bộ stage. Điều này cho phép jobs chạy song song và giảm thời gian pipeline.

07 — Docker

🐳 Dockerfile & Build Tối ưu

Dockerfile multi-stage cho Java Spring Boot để image nhỏ gọn và build nhanh hơn.

Dockerfile · multi-stage build
# ── Stage 1: Build ─────────────────────────────
FROM maven:3.9-eclipse-temurin-17-alpine AS builder
WORKDIR /app

# Cache dependencies layer riêng
COPY pom.xml .
RUN mvn dependency:go-offline -B

COPY src ./src
RUN mvn package -DskipTests -B

# ── Stage 2: Extract layers (Spring Boot) ──────
FROM eclipse-temurin:17-jre-alpine AS extractor
WORKDIR /app
COPY --from=builder /app/target/*.jar app.jar
RUN java -Djarmode=layertools -jar app.jar extract

# ── Stage 3: Runtime ───────────────────────────
FROM eclipse-temurin:17-jre-alpine
WORKDIR /app

# Non-root user for security
RUN addgroup -S appgroup && adduser -S appuser -G appgroup
USER appuser

# Copy layers (ordered từ ít thay đổi → nhiều thay đổi)
COPY --from=extractor /app/dependencies/ ./
COPY --from=extractor /app/spring-boot-loader/ ./
COPY --from=extractor /app/snapshot-dependencies/ ./
COPY --from=extractor /app/application/ ./

EXPOSE 8080
ENTRYPOINT ["java", "org.springframework.boot.loader.JarLauncher"]

📤 Push lên AWS ECR

YAML · publish stage
push-ecr:
  stage: publish
  image:
    name: amazon/aws-cli:latest
    entrypoint: [""]
  services:
    - docker:24-dind
  before_script:
    - yum install -y docker
    - aws ecr get-login-password --region $AWS_REGION |
        docker login --username AWS --password-stdin
        $AWS_ACCOUNT_ID.dkr.ecr.$AWS_REGION.amazonaws.com
  script:
    # Load image từ artifact
    - docker load < image.tar.gz
    # Ensure ECR repo exists
    - aws ecr describe-repositories --repository-names $CI_PROJECT_NAME
        || aws ecr create-repository --repository-name $CI_PROJECT_NAME
    # Push với 2 tags: commit SHA + branch slug
    - docker push $IMAGE_NAME:$IMAGE_TAG
    - docker tag $IMAGE_NAME:$IMAGE_TAG $IMAGE_NAME:$CI_COMMIT_REF_SLUG
    - docker push $IMAGE_NAME:$CI_COMMIT_REF_SLUG
  needs: [docker-build]
💡 GitLab Container Registry
Nếu không dùng AWS ECR, GitLab tích hợp sẵn Container Registry. Dùng $CI_REGISTRY, $CI_REGISTRY_USER, $CI_REGISTRY_PASSWORD để login và push trực tiếp.
08 — Cache

⚡ Cache & Artifacts

📦
Cache
Lưu files giữa các pipeline runs khác nhau. Dùng cho Maven .m2/repository, Node node_modules. Tăng tốc build đáng kể.
📎
Artifacts
Pass files giữa các jobs trong cùng pipeline. Dùng để truyền .jar từ build → publish, hoặc test reports cho GitLab UI.
YAML · cache strategy
# Global cache config
cache:
  key:
    files:
      - pom.xml         # Cache key thay đổi khi pom.xml thay đổi
    prefix: maven-v1
  paths:
    - .m2/repository/
  policy: pull-push   # download rồi upload lại nếu có thay đổi

# Job chỉ pull (không cần update cache)
unit-test:
  cache:
    policy: pull      # nhanh hơn vì không upload

# Job update cache (chạy 1 lần duy nhất)
maven-build:
  cache:
    policy: push      # chỉ update, không download
09 — Kubernetes

☸️ Kubernetes Manifests

Các K8s resources cần thiết cho một Java Spring Boot service.

📦 Deployment

YAML · deployment.yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: my-service
  namespace: production
  labels:
    app: my-service
spec:
  replicas: 3
  selector:
    matchLabels:
      app: my-service
  strategy:
    type: RollingUpdate
    rollingUpdate:
      maxSurge: 1        # tối đa 1 pod mới thêm
      maxUnavailable: 0  # không pod nào down trong update
  template:
    metadata:
      labels:
        app: my-service
    spec:
      containers:
        - name: my-service
          image: 123456789.dkr.ecr.ap-southeast-1.amazonaws.com/my-service:abc12345
          ports:
            - containerPort: 8080
          env:
            - name: SPRING_PROFILES_ACTIVE
              value: production
            - name: DB_PASSWORD
              valueFrom:
                secretKeyRef:
                  name: my-service-secret
                  key: db-password
          resources:
            requests:
              memory: "512Mi"
              cpu: "250m"
            limits:
              memory: "1Gi"
              cpu: "500m"
          livenessProbe:
            httpGet:
              path: /actuator/health/liveness
              port: 8080
            initialDelaySeconds: 60
            periodSeconds: 15
          readinessProbe:
            httpGet:
              path: /actuator/health/readiness
              port: 8080
            initialDelaySeconds: 30
            periodSeconds: 10
✅ Luôn có Liveness + Readiness Probe
Liveness: K8s restart pod nếu app bị hang. Readiness: K8s không route traffic đến pod chưa sẵn sàng. Spring Boot Actuator cung cấp cả 2 endpoints này sẵn.

11 — Deploy Job

🚀 Deploy Job trong CI/CD

Job deploy dùng kubectl + helm để apply lên K8s cluster.

YAML · deploy stages đầy đủ
# ── Shared deploy template ──────────────────────
.deploy-template: &deploy-template
  image:
    name: alpine/helm:3.14.0
    entrypoint: [""]
  before_script:
    - apk add --no-cache curl python3 py3-pip
    - pip install awscli --quiet
    # Setup kubectl từ KUBE_CONFIG variable
    - mkdir -p ~/.kube
    - echo "$KUBE_CONFIG" | base64 -d > ~/.kube/config
    - chmod 600 ~/.kube/config
    # Login ECR để K8s pull image
    - aws ecr get-login-password --region $AWS_REGION |
        helm registry login --username AWS --password-stdin
        $AWS_ACCOUNT_ID.dkr.ecr.$AWS_REGION.amazonaws.com

# ── Deploy Dev (tự động trên branch develop) ────
deploy-dev:
  stage: deploy-dev
  <<: *deploy-template
  environment:
    name: development
    url: https://dev.my-service.internal
  script:
    - helm upgrade --install $HELM_RELEASE_NAME ./helm
      --namespace dev
      --create-namespace
      --values ./helm/values.yaml
      --values ./helm/values-dev.yaml
      --set image.repository=$IMAGE_NAME
      --set image.tag=$IMAGE_TAG
      --atomic         # rollback tự động nếu fail
      --timeout 5m
      --wait
  rules:
    - if: '$CI_COMMIT_BRANCH == "develop"'

# ── Deploy Prod (manual approval) ───────────────
deploy-prod:
  stage: deploy-prod
  <<: *deploy-template
  environment:
    name: production
    url: https://my-service.internal
  script:
    - helm upgrade --install $HELM_RELEASE_NAME ./helm
      --namespace production
      --values ./helm/values.yaml
      --values ./helm/values-prod.yaml
      --set image.repository=$IMAGE_NAME
      --set image.tag=$IMAGE_TAG
      --atomic
      --timeout 10m
      --wait
  when: manual          # ← cần bấm tay trong GitLab UI
  rules:
    - if: '$CI_COMMIT_BRANCH == "main"'
✅ --atomic flag
Flag --atomic của Helm sẽ tự động rollback về version trước nếu deployment thất bại (pods không healthy trong timeout). Rất quan trọng cho production.
12 — AWS EKS

☁️ AWS EKS Setup

1
Tạo IAM User cho CI/CD
Tạo IAM User với policy cho phép push ECR và describe EKS. Không dùng root account.
2
Cấu hình aws-auth ConfigMap
Cho phép IAM User access K8s cluster bằng cách thêm vào aws-auth ConfigMap trong namespace kube-system.
3
Export KUBE_CONFIG
Chạy aws eks update-kubeconfig rồi base64 encode file ~/.kube/config, lưu vào GitLab CI Variable.
4
Tạo ECR Repository
Tạo ECR repo cho từng service. Configure lifecycle policy để xóa old images tự động.
Shell · setup commands
# 1. Update kubeconfig
aws eks update-kubeconfig \
  --region ap-southeast-1 \
  --name my-eks-cluster

# 2. Export KUBE_CONFIG (paste vào GitLab Variable)
cat ~/.kube/config | base64 -w 0

# 3. Thêm IAM vào aws-auth
kubectl edit configmap aws-auth -n kube-system
# Thêm vào mapUsers:
# - userarn: arn:aws:iam::123456789:user/gitlab-ci
#   username: gitlab-ci
#   groups: ["system:masters"]  # hoặc role cụ thể hơn

# 4. Tạo ECR repo
aws ecr create-repository \
  --repository-name my-service \
  --region ap-southeast-1

# 5. Set ECR lifecycle policy (giữ 10 images mới nhất)
aws ecr put-lifecycle-policy \
  --repository-name my-service \
  --lifecycle-policy-text '{"rules":[{"rulePriority":1,"selection":{"tagStatus":"any","countType":"imageCountMoreThan","countNumber":10},"action":{"type":"expire"}}]}'

  stage: deploy-staging
  environment:
    name: staging
    url: https://staging.my-service.internal

14 — Rollback

⏪ Rollback Strategy

🔴 Khi nào cần rollback?
Deployment fail (pods crash loop), health check fail, business metrics tụt đột ngột sau deploy mới.

1. Helm Rollback (nhanh nhất)

Shell · helm rollback
# Xem lịch sử releases
helm history my-service -n production

# Rollback về revision trước
helm rollback my-service -n production

# Rollback về revision cụ thể
helm rollback my-service 3 -n production

2. Rollback Job trong CI

YAML · rollback job
rollback-prod:
  stage: deploy-prod
  image: alpine/helm:3.14.0
  environment:
    name: production
    action: stop    # GitLab environment action
  script:
    - echo "$KUBE_CONFIG" | base64 -d > ~/.kube/config
    - helm rollback $HELM_RELEASE_NAME -n production
    - kubectl rollout status deployment/$HELM_RELEASE_NAME -n production
  when: manual
  rules:
    - if: '$CI_COMMIT_BRANCH == "main"'
✅ GitLab Environment Rollback
GitLab UI → Deployments → Environments → Production → có nút Rollback built-in, re-trigger pipeline với commit hash cũ hơn.

🔍 Smoke Test sau Deploy

YAML · smoke-test job
smoke-test:
  stage: deploy-dev
  image: curlimages/curl:latest
  needs: [deploy-dev]
  script:
    # Chờ service up (retry 10 lần)
    - for i in $(seq 1 10); do
    -   curl -f http://my-service.dev.svc.cluster.local/actuator/health
    -   && echo "✅ Service healthy" && break
    -   || echo "⏳ Waiting... ($i/10)" && sleep 15
    - done
    # Test endpoint chính
    - curl -f http://my-service.dev.svc.cluster.local/api/v1/ping
📈
Prometheus
Scrape metrics từ /actuator/prometheus. Setup ServiceMonitor nếu dùng kube-prometheus-stack.

💡 Tips, Tricks & Common Pitfalls

⚠️ Pitfall #1 — Image tag "latest"
Không bao giờ dùng :latest trong production. Dùng commit SHA để mỗi deploy có tag duy nhất, có thể trace và rollback chính xác.
⚠️ Pitfall #2 — initialDelaySeconds quá thấp
Spring Boot khởi động chậm (30-60s khi load beans, Liquibase migrations...). Set initialDelaySeconds đủ cao hoặc dùng startupProbe để tránh K8s restart pod liên tục.
⚠️ Pitfall #3 — Không set resource limits
Pods không có resource limits có thể eat toàn bộ node resources, ảnh hưởng các services khác. Luôn set cả requests và limits.
🔴 Pitfall #4 — Secrets trong logs
Tránh echo $SECRET_VAR trong script. GitLab mask variables nhưng nếu secret bị fragment thì vẫn có thể lộ. Dùng file thay vì env var cho sensitive data dài.

✅ Best Practices tổng hợp

  • Tag image bằng $CI_COMMIT_SHORT_SHA — traceability từ image → commit
  • Dùng needs: [] — giảm pipeline time bằng cách chạy jobs song song thay vì tuần tự theo stage
  • --atomic trong helm upgrade — auto rollback nếu deployment fail
  • Separate kubeconfig per environment — dev/staging/prod dùng kubeconfig riêng, không share
  • Protected + Masked variables cho prod secrets — không accessible từ non-protected branches
  • Cache key theo pom.xml checksum — tự invalidate khi deps thay đổi
  • artifacts: expire_in — không để artifacts tích tụ tốn storage
  • Limit Docker layer cache — COPY pom.xml trước rồi mvn dependency:go-offline để tận dụng Docker layer cache
  • Non-root user trong Docker — security best practice
  • startupProbe cho Spring Boot — tốt hơn initialDelaySeconds cứng nhắc

🛠 Debugging Commands

Shell · debugging K8s deploy
# Xem trạng thái pods
kubectl get pods -n production -l app=my-service

# Xem logs pod (follow)
kubectl logs -f deployment/my-service -n production

# Describe pod để xem events (crash reason)
kubectl describe pod  -n production

# Xem rollout status
kubectl rollout status deployment/my-service -n production

# Xem lịch sử Helm
helm history my-service -n production

# Xem values đang dùng
helm get values my-service -n production

# Dry-run helm upgrade (không apply thật)
helm upgrade --dry-run my-service ./helm \
  --set image.tag=abc12345 -n production

# Port-forward để test locally
kubectl port-forward svc/my-service 8080:80 -n production
