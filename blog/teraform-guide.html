OVERVIEW

Terraform là gì? — Infrastructure as Code

Tại sao cần IaC, cách Terraform hoạt động, so sánh với Ansible và CloudFormation

Vấn đề IaC giải quyết — Ví dụ thực tế
❌ Trước IaC — Manual Click
1. Vào AWS Console → tạo VPC
2. Tạo Subnets thủ công từng cái
3. Cấu hình Security Groups
4. Launch EC2, chờ, cấu hình
5. Lần sau làm lại từ đầu 😭
6. Staging ≠ Production → bugs
✅ Với Terraform — Declarative
1. Viết file .tf 1 lần
2. terraform apply → tạo tất cả
3. Lần sau: thay env = "staging"
4. apply lại → giống hệt production
5. Git history = lịch sử infrastructure
6. destroy → xóa sạch, không lo quên

WORKFLOW

Terraform Workflow — init, plan, apply, destroy

Quy trình làm việc hàng ngày với Terraform từ đầu đến cuối

Tất cả commands với giải thích
terraform commands — full workflowCLI
# ── Bước 1: terraform init ──
$ terraform init
Initializing the backend...
Initializing provider plugins...
- Finding hashicorp/aws versions matching "~> 5.0"...
- Installing hashicorp/aws v5.31.0...
Terraform has been successfully initialized!
# init: download providers, setup backend, install modules
# Chạy khi: lần đầu, thêm provider mới, thay đổi backend

# ── Bước 2: terraform validate ──
$ terraform validate
Success! The configuration is valid.
# Kiểm tra syntax HCL mà không gọi API

# ── Bước 3: terraform fmt ──
$ terraform fmt -recursive  # format code đẹp
$ terraform fmt -check       # chỉ check, không sửa (dùng trong CI)

# ── Bước 4: terraform plan ──
$ terraform plan
Terraform will perform the following actions:

  # aws_instance.web will be created
  + resource "aws_instance" "web" {
      + ami           = "ami-0c55b159cbfafe1f0"
      + instance_type = "t3.micro"
    }

Plan: 1 to add, 0 to change, 0 to destroy.

$ terraform plan -out=tfplan    # lưu plan vào file (dùng trong CI/CD)
$ terraform plan -var-file=prod.tfvars

# ── Bước 5: terraform apply ──
$ terraform apply              # hỏi yes/no
$ terraform apply -auto-approve  # không hỏi (dùng trong CI)
$ terraform apply tfplan        # dùng plan đã save

# Apply chỉ 1 resource cụ thể
$ terraform apply -target=aws_instance.web

# ── Các commands khác ──
$ terraform show              # xem state hiện tại
$ terraform state list        # list tất cả resources trong state
$ terraform state show aws_instance.web  # chi tiết 1 resource
$ terraform output            # xem output values
$ terraform output vpc_id     # output cụ thể
$ terraform refresh           # sync state với real world
$ terraform destroy           # XÓA tất cả resources!
$ terraform graph | dot -Tpng > graph.png  # visualize dependency

resource "aws_vpc" "main" {
  cidr_block           = "10.0.0.0/16"
  enable_dns_hostnames = true
  enable_dns_support   = true

  tags = {
    Name        = "main-vpc"
    Environment = var.environment  # reference variable
    ManagedBy   = "terraform"
  }
}

resource "aws_subnet" "public" {
  vpc_id            = aws_vpc.main.id  # reference resource trên
  cidr_block        = "10.0.1.0/24"
  availability_zone = "ap-southeast-1a"

  tags = {
    Name = "public-subnet"
  }
}

resource "aws_instance" "web" {
  ami           = data.aws_ami.ubuntu.id  # từ data source
  instance_type = var.instance_type
  subnet_id     = aws_subnet.public.id

  root_block_device {               # nested block
    volume_type = "gp3"
    volume_size = 20
    encrypted   = true
  }

  tags = {
    Name = "web-server"
  }

  lifecycle {                         # lifecycle meta-argument
    create_before_destroy = true     # tạo mới trước khi xóa cũ
    prevent_destroy       = false    # true = bảo vệ khỏi destroy
    ignore_changes        = [tags]   # bỏ qua thay đổi tags từ ngoài
  }

  depends_on = [aws_internet_gateway.main]  # explicit dependency
}

Variables & Outputs — Input & Output

Biến đầu vào linh hoạt, output để chia sẻ giá trị giữa modules

Variables — khai báo và sử dụng
variables.tf — tất cả loại variableVariables
# ── string variable ──
variable "environment" {
  description = "Environment name (dev/staging/prod)"
  type        = string
  default     = "dev"

  validation {                    # validate input
    condition     = contains(["dev", "staging", "prod"], var.environment)
    error_message = "Environment must be dev, staging, or prod."
  }
}

# ── number variable ──
variable "instance_count" {
  description = "Number of EC2 instances"
  type        = number
  default     = 2
}

# ── list variable ──
variable "availability_zones" {
  type    = list(string)
  default = ["ap-southeast-1a", "ap-southeast-1b"]
}

# ── map variable ──
variable "instance_types" {
  type = map(string)
  default = {
    dev     = "t3.micro"
    staging = "t3.small"
    prod    = "t3.medium"
  }
}
# Dùng: var.instance_types[var.environment] → "t3.micro"

# ── object variable (complex type) ──
variable "database_config" {
  type = object({
    engine         = string
    engine_version = string
    instance_class = string
    storage_gb     = number
    multi_az       = bool
  })
  default = {
    engine         = "postgres"
    engine_version = "15.4"
    instance_class = "db.t3.micro"
    storage_gb     = 20
    multi_az       = false
  }
}

# ── sensitive variable (không hiện trong logs) ──
variable "db_password" {
  type      = string
  sensitive = true   # giá trị bị ẩn trong plan/apply output
}

# ── Cách truyền giá trị vào variable ──
# 1. terraform.tfvars:
environment    = "prod"
instance_count = 5

# 2. Command line:
$ terraform plan -var="environment=prod" -var="instance_count=5"

# 3. Environment variables (prefix TF_VAR_):
$ TF_VAR_environment=prod TF_VAR_db_password=secret terraform apply

LOCALS

Locals & Expressions — Biến tính toán và biểu thức

locals tránh lặp code · expressions xử lý dữ liệu · functions built-in

locals và Terraform functions phổ biến
locals.tf — computed values + built-in functionsLocals
locals {
  # Computed từ variables
  is_production  = var.environment == "prod"
  name_prefix    = "${var.project}-${var.environment}"
  instance_type  = var.instance_types[var.environment]

  # Common tags — dùng lại khắp nơi
  common_tags = {
    Project     = var.project
    Environment = var.environment
    ManagedBy   = "terraform"
    Repository  = "github.com/company/infra"
  }

  # String functions
  bucket_name  = lower("${var.project}-${var.environment}-assets")
  service_name = replace(var.project, "_", "-")

  # List/Map functions
  azs          = slice(data.aws_availability_zones.available.names, 0, 3)
  merged_tags  = merge(local.common_tags, { Team = "backend" })
  subnet_cidrs = cidrsubnets("10.0.0.0/16", 8, 8, 8)  # auto-calculate CIDRs

  # Conditional
  db_instance_class = local.is_production ? "db.t3.medium" : "db.t3.micro"
  multi_az          = local.is_production
}

# Dùng local:
resource "aws_s3_bucket" "assets" {
  bucket = local.bucket_name
  tags   = local.common_tags
}

# ── Các functions hay dùng ──
format("%s-%s", var.env, "web")         # "prod-web"
toset(["a", "b", "a"])                  # {"a", "b"}
tolist(local.some_set)                   # convert set → list
jsonencode(local.common_tags)            # → JSON string
filebase64("./scripts/init.sh")          # encode file → base64
templatefile("tpl/user_data.tftpl", {     # render template
  db_host = aws_db_instance.main.endpoint
})
lookup(var.instance_types, var.env, "t3.micro")  # map lookup với default
length(var.availability_zones)             # đếm items
flatten([["a", "b"], ["c"]])              # → ["a", "b", "c"]
distinct(["a", "b", "a"])                # → ["a", "b"]
LOOPS

count, for_each, dynamic — Tạo nhiều resources

Tránh lặp code — một khai báo tạo N resources với giá trị khác nhau

count vs for_each — chọn cái nào?
count + for_each + for expression + dynamic blockLoops
# ── count — tạo N resources giống nhau ──
resource "aws_instance" "web" {
  count         = var.instance_count  # 3 → tạo 3 instances
  ami           = data.aws_ami.ubuntu.id
  instance_type = "t3.micro"

  tags = {
    Name = "web-${count.index}"   # web-0, web-1, web-2
  }
}
# Tham chiếu: aws_instance.web[0].id, aws_instance.web[*].id
# Nhược điểm: xóa web[0] → web[1] trở thành web[0] → recreate!

# ── for_each — tạo resources từ map/set (ổn định hơn count) ──
variable "subnets" {
  type = map(object({
    cidr = string
    az   = string
  }))
  default = {
    public-1a  = { cidr = "10.0.1.0/24", az = "ap-southeast-1a" }
    public-1b  = { cidr = "10.0.2.0/24", az = "ap-southeast-1b" }
    private-1a = { cidr = "10.0.10.0/24", az = "ap-southeast-1a" }
  }
}

resource "aws_subnet" "all" {
  for_each = var.subnets           # each.key = "public-1a", each.value = {cidr,az}

  vpc_id            = aws_vpc.main.id
  cidr_block        = each.value.cidr
  availability_zone = each.value.az

  tags = {
    Name = "subnet-${each.key}"
  }
}
# Tham chiếu: aws_subnet.all["public-1a"].id
# Stable: xóa "public-1a" không ảnh hưởng "public-1b"!

# ── for expression — transform collections ──
locals {
  # List comprehension
  public_subnet_ids  = [for k, v in aws_subnet.all : v.id if startswith(k, "public")]

  # Map comprehension
  subnet_name_to_id  = {for k, v in aws_subnet.all : k => v.id}
}

# ── dynamic block — tạo nested blocks lặp ──
resource "aws_security_group" "app" {
  name   = "app-sg"
  vpc_id = aws_vpc.main.id

  dynamic "ingress" {
    for_each = [80, 443, 8080]   # tạo 3 ingress rules
    content {
      from_port   = ingress.value
      to_port     = ingress.value
      protocol    = "tcp"
      cidr_blocks = ["0.0.0.0/0"]
    }
  }
}

Providers & Versions — Cấu hình Provider

AWS/GCP/Azure providers, version constraints, multiple accounts

providers.tf và versions.tf
versions.tf + providers.tf — multi-region AWSProviders
# ── versions.tf ──
terraform {
  required_version = ">= 1.6.0"   # minimum Terraform version

  required_providers {
    aws = {
      source  = "hashicorp/aws"
      version = "~> 5.0"       # >= 5.0.0, < 6.0.0
    }
    kubernetes = {
      source  = "hashicorp/kubernetes"
      version = ">= 2.23.0"
    }
    helm = {
      source  = "hashicorp/helm"
      version = ">= 2.12.0"
    }
  }

  backend "s3" {               # remote backend
    bucket         = "company-terraform-states"
    key            = "prod/main.tfstate"
    region         = "ap-southeast-1"
    encrypt        = true
    dynamodb_table = "terraform-state-lock"
  }
}

# ── providers.tf ──
provider "aws" {
  region  = var.aws_region
  profile = var.aws_profile  # ~/.aws/credentials

  default_tags {
    tags = local.common_tags  # áp dụng cho TẤT CẢ resources!
  }

  # Assume Role (CI/CD, cross-account)
  assume_role {
    role_arn = "arn:aws:iam::123456789:role/TerraformRole"
  }
}

# ── Multiple providers: deploy sang nhiều regions ──
provider "aws" {
  alias  = "us_east"           # alias để phân biệt
  region = "us-east-1"
}

provider "aws" {
  alias  = "ap_southeast"
  region = "ap-southeast-1"
}

# Dùng với alias:
resource "aws_s3_bucket" "logs_us" {
  provider = aws.us_east      # tạo ở us-east-1
  bucket   = "logs-us-east"
}

# ── Version constraints ──
# "5.0.0"   — exactly 5.0.0
# "~> 5.0"  — >= 5.0, < 6.0 (patch level)
# "~> 5.1.0"— >= 5.1.0, < 5.2.0 (bug fixes)
# ">= 5.0, < 6.0" — explicit range

State File & Remote Backend

Tại sao cần remote backend, cấu hình S3 + DynamoDB cho team

State file là gì và tại sao phải dùng Remote Backend?
💡 Giải thích
terraform.tfstate — Mapping giữa code và reality
State file lưu thông tin về tất cả resources đã tạo: ID, IP, ARN... Khi terraform plan, nó so sánh state với code và với AWS để tính ra thay đổi cần thực hiện.

Vấn đề với local state: Mỗi người có state riêng → conflict khi làm team. Giải pháp: Remote backend (S3 + DynamoDB) → tất cả cùng dùng 1 state, có lock.
bootstrap/main.tf — tạo S3 backend + DynamoDB lock tableRemote Backend
# ── Bước 1: tạo S3 bucket để lưu state (chỉ cần làm 1 lần) ──
resource "aws_s3_bucket" "terraform_state" {
  bucket = "company-terraform-states"

  lifecycle {
    prevent_destroy = true   # KHÔNG cho xóa bucket này!
  }
}

resource "aws_s3_bucket_versioning" "terraform_state" {
  bucket = aws_s3_bucket.terraform_state.id
  versioning_configuration {
    status = "Enabled"   # versioning để rollback state
  }
}

resource "aws_s3_bucket_server_side_encryption_configuration" "state" {
  bucket = aws_s3_bucket.terraform_state.id
  rule {
    apply_server_side_encryption_by_default {
      sse_algorithm = "AES256"
    }
  }
}

resource "aws_s3_bucket_public_access_block" "state" {
  bucket                  = aws_s3_bucket.terraform_state.id
  block_public_acls       = true
  block_public_policy     = true
  ignore_public_acls      = true
  restrict_public_buckets = true
}

# ── Bước 2: DynamoDB table để lock state ──
resource "aws_dynamodb_table" "terraform_lock" {
  name         = "terraform-state-lock"
  billing_mode = "PAY_PER_REQUEST"
  hash_key     = "LockID"

  attribute {
    name = "LockID"
    type = "S"
  }

  lifecycle {
    prevent_destroy = true
  }
}

# ── Bước 3: Cấu hình backend trong versions.tf ──
terraform {
  backend "s3" {
    bucket         = "company-terraform-states"
    key            = "production/eks/terraform.tfstate"  # path trong bucket
    region         = "ap-southeast-1"
    encrypt        = true
    dynamodb_table = "terraform-state-lock"
  }
}

STATE OPS

State Operations — Import, Move, Remove

Xử lý state khi có resources tạo ngoài Terraform hoặc cần refactor

Thao tác state nâng cao
State operations — import, mv, rm, workspaceState
# ── terraform import — đưa resource tạo ngoài vào state ──
# Tình huống: AI có EC2 i-0abc1234 tạo thủ công, muốn quản lý bằng TF
# Bước 1: viết resource block trong code
resource "aws_instance" "existing" {
  # ... điền attributes sau khi import
}
# Bước 2: import
$ terraform import aws_instance.existing i-0abc1234def56789
aws_instance.existing: Importing from ID "i-0abc1234def56789"...
Import successful!
# Bước 3: terraform show → copy attributes vào code

# Khai báo import trong code (Terraform 1.5+) — cleaner!
import {
  id = "i-0abc1234def56789"
  to = aws_instance.existing
}
# terraform plan → tự generate resource code
$ terraform plan -generate-config-out=generated.tf

# ── terraform state mv — rename/move resource trong state ──
# Đổi tên resource trong code mà không destroy+recreate
$ terraform state mv aws_instance.old_name aws_instance.new_name
$ terraform state mv aws_subnet.public 'module.network.aws_subnet.public'

# ── moved block trong code (Terraform 1.1+) ──
moved {
  from = aws_instance.old
  to   = aws_instance.new
}

# .gitlab-ci.yml
image:
  name: hashicorp/terraform:1.6.0
  entrypoint: [""]

variables:
  TF_ROOT:         ${CI_PROJECT_DIR}/environments/prod
  TF_STATE_NAME:   production
  TF_IN_AUTOMATION:"true"
  AWS_DEFAULT_REGION: ap-southeast-1

cache:
  key:   "${TF_ROOT}"
  paths: ["${TF_ROOT}/.terraform"]

before_script:
  - cd ${TF_ROOT}
  - terraform init -backend-config=backend.hcl -input=false

stages: [validate, plan, apply]

validate:
  stage: validate
  script:
    - terraform validate
    - terraform fmt -check -recursive

plan:
  stage: plan
  script:
    - terraform plan -var-file=terraform.tfvars -out=tfplan -input=false
    - terraform show -no-color tfplan > plan.txt
  artifacts:
    paths:   ["${TF_ROOT}/tfplan"]
    reports:
      terraform: "${TF_ROOT}/plan.txt"   # hiện plan trong MR!
    expire_in: 1 week

apply:
  stage:  apply
  when:   manual                          # click tay để deploy
  script:
    - terraform apply -input=false tfplan
  environment:
    name: production
  rules:
    - if: $CI_COMMIT_BRANCH == $CI_DEFAULT_BRANCH
      when: manual
    - when: never
  needs: [plan]
  resource_group: production              # chỉ 1 apply cùng lúc!

Lộ trình học Terraform — từ zero đến production
1

Install và chạy Hello World

Cài Terraform CLI. Viết main.tf tạo S3 bucket. Chạy init → plan → apply → destroy. Hiểu luồng cơ bản.

2

Nắm HCL: variables, outputs, locals

Viết variables.tf với nhiều loại type. Dùng locals để tránh lặp. Outputs để export values. Viết terraform.tfvars cho từng env.

3

Setup Remote Backend (S3 + DynamoDB)

Tạo S3 bucket và DynamoDB table. Cấu hình backend "s3". Chạy terraform init -migrate-state. Kiểm tra state đã lên S3.

4

Viết Module đầu tiên

Tạo folder modules/networking/ với VPC, subnets, route tables. Gọi từ environments/dev/main.tf. Hiểu module.xxx.output pattern.

5

Multi-environment với separate directories

Cấu trúc environments/dev, environments/prod. Dùng chung modules nhưng khác terraform.tfvars. Backend key khác nhau. Hiểu tại sao không dùng workspace.
